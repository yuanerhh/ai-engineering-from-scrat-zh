# DualPipe 并行

> DeepSeek-V3 在 2048 块 H800 GPU 上训练，MoE 专家分散在各节点上。跨节点专家 all-to-all 通信每消耗 1 GPU 小时的计算就要花 1 GPU 小时的通信，GPU 有一半时间在闲置。DualPipe（DeepSeek，2024 年 12 月）是一种双向流水线，将前向和反向计算与它们触发的 all-to-all 通信重叠起来。气泡减少，吞吐量提升，而保持两份模型参数副本（"双"这个名字的由来）的代价很低，因为专家并行本来就已经把专家分散到各 rank 了。本课学习 DualPipe 实际做了什么，以及为何 Sea AI Lab 的 DualPipeV 改进以略微紧张的气泡为代价消除了 2 倍参数代价。

**类型：** 学习
**语言：** Python（标准库，调度模拟器）
**前置条件：** Phase 10 · 第05课（分布式训练，FSDP，DeepSpeed），Phase 10 · 第14课（开放模型架构和 MoE）
**时间：** 约 60 分钟

## 学习目标

- 列举 DualPipe 前向-反向分块的四个组件，以及为何每个组件都有自己的重叠窗口
- 解释规模化时的流水线气泡问题，以及"无气泡"在实践中与营销宣传上各意味着什么
- 手动追踪 8 个 PP rank、16 个微批次的 DualPipe 调度，确认前向和反向流如何填充对方的空闲槽
- 阐述 DualPipeV（Sea AI Lab，2025）的权衡：在专家并行未激活时以略大的气泡为代价去掉 2 倍参数复制

## 问题所在

在 2000 块 H800 GPU 上训练 6710 亿参数的 MoE 模型会遇到三个叠加的瓶颈：

1. **内存压力：** 每块 GPU 持有模型的一个切片。在 61 层、128 个注意力头上以序列长度 8k 计算的激活内存是巨大的。
2. **流水线气泡：** 传统流水线并行（GPipe、1F1B）在等待本阶段输入或梯度时让 GPU 闲置。在 8 个阶段时，即使使用 1F1B 调度，气泡也可能占 GPU 时间的约 12%。
3. **跨节点 all-to-all：** 带专家并行的 MoE 将专家分散到各节点，每次前向传播都会触发一次 all-to-all 将词元发送到专家，再来一次汇聚。在 2000 块 GPU 时，这很容易变成 1:1 的计算通信比。

这些各有独立解决方案：梯度检查点用于内存，Zero Bubble（Sea AI Lab，2023）用于流水线气泡，专家并行通信内核用于 all-to-all。DualPipe 的作用是让它们协同工作。该调度在单个前向-反向分块内重叠计算和通信，从流水线两端同时注入微批次，并利用由此产生的调度将 all-to-all 隐藏在计算窗口内。

报告结果：流水线气泡近乎消除，DeepSeek-V3 的 14.8 万亿词元训练中 GPU 利用率超过 95%。

## 概念讲解

### 流水线并行回顾

将 N 层模型拆分到 P 个设备上。设备 `i` 持有层 `i * N/P .. (i+1) * N/P - 1`。一个微批次从设备 0 到 P-1 前向传播，然后从 P-1 到 0 反向传播。每个设备只有在前一个设备发来输出后才能开始前向传播，只有在下游设备发来上游梯度后才能开始反向传播。

GPipe（Huang 等人，2019）一次调度一个微批次，浪费了大部分 GPU 时间。1F1B（Narayanan 等人，2021）交错多个微批次的前向和反向传播。Zero Bubble（Qi 等人，2023）将反向传播分成两部分——输入反向（B）和权重反向（W）——并调度它们填充气泡。经过 Zero Bubble 后，流水线几乎紧密。

DualPipe 是下一步，在此基础上增加了两个思路：

### 思路一：分块分解

每个前向分块分成四个组件：

- **注意力：** Q/K/V 投影、注意力、输出投影
- **All-to-all 分发：** 将词元发送到各专家的跨节点通信
- **MLP：** MoE 专家计算
- **All-to-all 汇聚：** 将专家输出带回的跨节点通信

反向分块添加各自对应的梯度版本。DualPipe 调度使 all-to-all 分发与下一分块的注意力计算并行，all-to-all 汇聚与后一分块的 MLP 计算并行。

### 思路二：双向调度

大多数流水线调度从阶段 0 注入微批次并流向阶段 P-1。DualPipe 从**两端**同时注入微批次。阶段 0 看到从这里发起的前向微批次，阶段 P-1 也看到从那里发起的前向微批次，两个流在中间相遇。

为使这一切正常工作，设备 `i` 必须同时持有早期流水线层 `i` 和后期流水线层 `P - 1 - i`。这就是 DualPipe 中"双"的由来：每个设备保存两份它需要服务的模型层（每个方向一份）。在 DeepSeek-V3 的规模下，这是 2 倍参数复制代价，但因为专家并行已经将 MoE 专家分散得很稀疏，将非专家层复制两份相对而言代价很小。

关键是：一个方向的前向流和另一个方向的反向流恰好在单向调度会有气泡的地方重叠，气泡消失了。

### 手动追踪调度

考虑 P=4 个 rank，8 个微批次，分为 4 个前向/4 个反向。时间从左到右，行是设备 rank：

```
           时间 →
rank 0:  F1 F2 F3 F4  F5R F6R F7R F8R  B1 B2 B3 B4  ...
rank 1:     F1 F2 F3  F4/F5R F6R F7R   B1 B2 ...
rank 2:        F1 F2  F3/F5R F4/F6R    B1 ...
rank 3:           F1  F2/F5R F3/F6R    ...
```

解读"F4/F5R"记号：rank 1 在同一时间槽内**既在运行**微批次 4 的前向传播（在流水线中从左到右），**也在运行**微批次 5 的前向传播（从右到左）。这在操作层面上就是"双向"的含义。

在 rank 2 处，两个流更早重叠；在 rank 0 和 P-1 处，重叠最晚。在调度的稳定中间阶段，每个 rank 运行 X 方向的前向与 Y 方向的反向重叠。计算持续忙碌，前向传播的 all-to-all 分发隐藏在反向计算内，all-to-all 汇聚隐藏在前向计算内，气泡被挤出。

### 气泡统计

标准 1F1B 流水线气泡（每个 rank 浪费的时间）：

```
bubble_1F1B = (P - 1) * forward_chunk_time
```

Zero Bubble 改进降低了这个数字但没到零。DualPipe 在稳定阶段，当微批次数量是流水线深度两倍的整数倍时气泡为零。在稳定阶段之外（预热和冷却），有一些气泡，但不随微批次数量增长——这是论文强调的关键特性。

营销表述："无气泡"。技术表述：气泡不随微批次数量增长。Sea AI Lab 的后续分析（DualPipeV / 一分为二）表明，只有在专家并行不是瓶颈时才能实现完全的零气泡；当有 EP 驱动的 all-to-all 时，总是存在一些调度妥协。

### DualPipeV——改进版

Sea AI Lab（2025）观察到，当 EP 通信重叠不是重点时，2 倍参数复制是浪费的。他们的 DualPipeV 调度将双向注入折叠成一个"V 形"调度，在单份参数副本上运行。气泡比 DualPipe 略大，但内存节省是实质性的。DeepSeek 在其开源 DualPipe 实现中采用 DualPipeV 作为 EP 关闭模式。

权衡对比：

| 特性 | DualPipe | DualPipeV | 1F1B | Zero Bubble |
|-----|---------|-----------|------|------------|
| 每设备参数副本数 | 2 | 1 | 1 | 1 |
| 气泡随微批次 | 恒定 | 小幅增长 | 增长 | 增长 |
| 计算通信重叠 | 完全 | 部分 | 极少 | 部分 |
| 适用场景 | EP 密集 MoE | 密集或 EP 轻量 | 基线 | 任何流水线 |

### 对 14.8 万亿词元训练的意义

DeepSeek-V3 的预训练在 2048 块 H800 GPU 上消耗了 14.8 万亿词元，约 280 万 GPU 小时。使用朴素 1F1B，流水线气泡会损失其中的 12-15%——34-42 万 GPU 小时，足以训练一个完整的 70B 模型。DualPipe 大部分地弥补了这一损失。在没有内部日志的情况下难以直接量化贡献，但论文声称整个训练的平均 GPU 利用率超过 95%。

对于规模较小的训练（1000 块 GPU 以下），DualPipe 是过度设计——流水线气泡相对于总代价较小，密集模型训练很少触及 all-to-all 瓶颈。对于数千块 GPU 规模的前沿 MoE 训练，它实际上是必须的。

### 在技术栈中的位置

- 与 **FSDP**（第05课）互补——FSDP 在 rank 间分片模型参数，DualPipe 在 rank 间调度计算，两者可以结合
- 兼容 **ZeRO-3** 梯度分片——两份副本复制的记账需要与 ZeRO 的分片梯度协作
- 需要针对特定集群拓扑调优的**自定义 all-to-all 内核**，DeepSeek 的开源内核是参考实现

## 实际运用

`code/main.py` 是一个流水线调度模拟器，接受 `(P, n_micro_batches, schedule)` 并打印 1F1B、Zero Bubble、DualPipe 和 DualPipeV 各自的稳定阶段利用率。这是一个教学工具——数字与论文中的定性声明一致，并非对生产测量加速比的声明。

模拟器的价值：用不同的 P 和微批次数量运行，观察 1F1B 的气泡比例如何增长而 DualPipe 不会。

实际训练运行的集成注意事项：

- 选择能整除微批次数量的流水线并行深度
- 确保你的专家并行网格支持双向 all-to-all，DeepSeek 的内核是参考
- 首次使用时预期在调度本身上花费一周调试时间，记账工作很繁琐
- 监控每个 rank 的 GPU 利用率而非仅看总量，DualPipe 的收益来自收紧落后者

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|---------|
| 流水线气泡 | "每个 rank 的闲置时间" | 流水线阶段在等待输入或梯度时浪费的 GPU 周期 |
| 1F1B | "默认流水线调度" | 一个前向/一个反向交错调度；DualPipe 要超越的基线 |
| Zero Bubble | "Sea AI Lab 2023" | 将反向拆分为 B（输入梯度）和 W（权重梯度）；几乎完全收紧流水线 |
| DualPipe | "DeepSeek-V3 调度" | 双向流水线+计算通信重叠；气泡不随微批次数量增长 |
| DualPipeV | "一分为二" | V 形改进，以略大的气泡为代价去掉 2 倍参数复制 |
| 分块 | "流水线工作单元" | 一个微批次通过一个流水线阶段的前向或反向传播 |
| All-to-all 分发 | "将词元发送到专家" | 将词元路由到其分配的 MoE 专家的跨节点通信 |
| All-to-all 汇聚 | "将专家输出带回" | MLP 后收集专家输出的跨节点通信 |
| 专家并行（EP）| "跨 GPU 的专家" | 将 MoE 专家跨 rank 分片，不同 GPU 持有不同专家 |
| 流水线并行（PP）| "跨 GPU 的层" | 将模型层跨 rank 分片；DualPipe 调度的维度 |
| 气泡比例 | "浪费的 GPU 时间" | (气泡时间 / 总时间)；DualPipe 驱向零的比例 |
