- 根据接受率 `α` 和草稿与验证器的代价比 `c` 计算预期加速比，并为每种场景选择最优草稿长度 `N`

## 问题所在

在 H100 上对 70B 模型进行自回归解码，速度大约只有每秒 35 个词元，GPU 远未饱和。内存带宽是瓶颈：每生成一个词元，就要从 HBM 加载 700 亿个权重，做一步算术，产生一个浮点数，而计算单元大部分时间是闲置的。

投机解码将这个问题转化为一个可以真正解决的吞吐量问题。廉价的草稿模型通过 `N` 次小型前向传播提出 `N` 个词元，验证器对前缀加所有 `N` 个草稿词元运行一次。如果验证器在位置 `i` 处的分布（在我们即将精确定义的统计意义上）与草稿一致，则接受；否则拒绝并从残差分布中采样修正。一次大模型前向传播最多产生 `N+1` 个被接受的词元，而非一个。

关键定理来自 Leviathan、Kalman、Matias（ICML 2023）：输出分布与直接从验证器采样产生的结果**完全一致**，不是近似，而是精确一致。这正是投机解码在生产中可以被接受的全部原因——它是纯粹的延迟优化，没有任何质量损失。

Phase 7 · 第16课给了你数学，本课给你训练栈。好的草稿模型比廉价草稿多带来 2× 的加速。EAGLE、EAGLE-2 和 EAGLE-3（Li 等人，2024-2025）将"草稿 = 同族的小模型"变成了一门精确的工程学科。2026 年的生产推理服务器默认使用 EAGLE-3。

## 概念讲解

### 不变量：Leviathan 拒绝采样

设 `p(t)` 为草稿在某个前缀下的下一词元分布，`q(t)` 为验证器的。从 `p` 中采样草稿词元 `d`，以概率 `min(1, q(d) / p(d))` 接受。若拒绝，则从残差分布 `(q - p)_+ / ||(q - p)_+||_1` 中采样。最终样本按 `q` 分布。这与 `p` 有多差无关——`p` 越差，拒绝越频繁，但输出始终精确。

用验证器的一次前向传播对 `prefix + d_1 + ... + d_N` 叠加 `N` 次这样的调用。验证器同时返回 `q_1, q_2, ..., q_{N+1}`。从左到右逐位置判断，在位置 `j` 首次拒绝时，从 `residual(q_j, p_j)` 采样并停止。若全部接受，则从 `q_{N+1}` 采样一个奖励词元。

### 决定加速比的因素

设 `α` 为每个草稿词元的预期接受率，`c = cost(draft) / cost(verifier)` 为代价比。每次验证器前向传播中预期被接受的词元数为：

```
E[accepted] = (1 - α^(N+1)) / (1 - α)
```

每个被接受词元的预期总墙上时间为 `(N * c + 1) / E[accepted]`。对 `N` 最小化即得甜蜜点。当 `α = 0.8, c = 0.05` 时，最优 `N` 约为 5-7，加速比为 3.2×；当 `α = 0.95, c = 0.02` 时，最优 `N` 约为 8-10，加速比接近 5×。

最大的单一杠杆是 `α`。在固定 `N = 5` 下，从 `α = 0.6`（朴素草稿）到 `α = 0.9`（EAGLE-3），每次验证器前向传播的预期被接受词元从 2.2 提升到 4.1，相当于接近 2× 的额外吞吐量，而验证器本身不变。

### 两年演进历程

**朴素投机解码（Leviathan，2023）：** 草稿模型是同族独立训练的小型 LLM。易于接入，`α ≈ 0.6`，加速比最多约 2×。

**EAGLE-1（Li 等人，2024）：** 草稿是一个微型 Transformer——通常只有一两层——以验证器最后一层的隐藏状态为输入，直接预测下一个词元。由于草稿能看到验证器的特征表示，其分布与验证器更为接近，`α` 提升到 0.7-0.8。

**EAGLE-2（Li 等人，2024）：** 添加了动态草稿树：不再提出一个单一的 `N` 词元序列，而是提出一棵候选树，用树形注意力在一次验证器前向传播中对所有候选打分，走概率最高的路径。草稿长度自适应调整，每个被接受路径词元的 `α` 超过 0.85。

**EAGLE-3（Li 等人，2025，NeurIPS）：** 两项进一步改变。第一，彻底去掉特征预测损失——EAGLE-1/2 训练草稿去匹配验证器的隐藏状态，这限制了数据量的帮助上限；EAGLE-3 直接在词元预测上训练。第二，训练时测试（TTT）：在草稿训练期间，在多个步骤上将草稿自身的先前预测作为输入反馈，与推理时的运行方式完全相同。这对齐了训练和测试分布，阻止了误差累积。实测加速比：对话场景最高 6.5×，在 SGLang 的 H100 上 batch 64 时吞吐量提升 38%。

### KV 缓存回滚

验证过程在一次传播中将验证器的 KV 缓存扩展 `N` 个条目。若在位置 `j` 发生拒绝，则位置 `j-1` 之后的缓存内容是错误的。两种常见实现：写入临时缓冲区，接受后提交（vLLM、TensorRT-LLM）；或维护物理 KV 缓存加逻辑长度，拒绝时截断。无论哪种方式，回滚代价是每层每头的字节数，相比前向传播代价可以忽略不计。

对于 EAGLE-2 的树形搜索，验证器使用尊重树拓扑的非因果掩码运行注意力。工程实现略显复杂，但计算本质上是带自定义掩码的标准 flash-attention 调用。

### 2026 年的草稿架构

| 策略 | 草稿类型 | `α` | 加速比 | 训练代价 |
|------|---------|-----|--------|---------|
| 朴素 | 独立小型 LLM | 0.55-0.70 | 1.8-2.3× | 无（复用已有小模型）|

## 动手实践

见 `code/main.py`。这是包含所有组件的完整 Leviathan 投机循环：起草 N 个词元、验证器并行传播、逐位置拒绝采样、残差采样、奖励词元、KV 回滚，以及验证输出分布与直接从 `q` 采样一致的经验验证。

### 第 1 步：拒绝规则

```python
def accept(q_prob, p_prob, u):
    if p_prob <= 0:
        return True
    return u < min(1.0, q_prob / p_prob)
```

### 第 2 步：残差分布

```python
def residual(q, p):
    raw = [max(0.0, qi - pi) for qi, pi in zip(q, p)]
    s = sum(raw)
    if s == 0:
        return list(q)
    return [r / s for r in raw]
```

### 第 3 步：完整的投机步骤

`spec_step` 函数从 `p` 中起草 `N` 个词元，然后在一次并行 `q` 评估中验证所有词元。对每个草稿词元应用拒绝规则，在首次拒绝时从残差中采样修正。若全部接受，则从 `q_{N+1}` 生成一个奖励词元。

### 第 4 步：KV 回滚记账

模拟器为每个工作进程追踪逻辑 `kv_length`。接受 `k` 个草稿时，`kv_length += k`。在位置 `j` 拒绝时，缓存已写入 `j` 之后的位置，但逻辑长度被设为 `prefix_length + j + 1`——即修正词元之后的一个位置。后续读取将截断到逻辑长度。

### 第 5 步：Leviathan 检验

运行 50,000 次投机步骤，统计被接受词元的经验分布，与 50,000 次直接从 `q` 采样的结果对比。卡方统计量应远低于临界值。定理在实践中成立。

### 第 6 步：加速比 vs. α

通过以不同幅度扰动 `p` 使其偏离 `q` 来扫描草稿质量，测量 `α`，然后绘制预期词元数与验证器调用次数的关系（作为 `α` 和 `N` 的函数）。代码打印表格，显示 EAGLE-3 级别的草稿质量（`α ≈ 0.9`）如何解锁每次验证器调用 4-5 个词元。

搭载 EAGLE-3 的生产级 `vllm serve`：

```bash
vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --speculative-config '{
    "model": "yuhuili/EAGLE3-LLaMA3.3-Instruct-70B",
    "num_speculative_tokens": 5,
    "method": "eagle3"
  }'
```

在 H100 上以 batch 64 运行 SGLang + EAGLE-3：根据 EAGLE-3 论文，相比 batch 64 朴素解码，吞吐量大约提升 1.38×。

适合使用投机解码的场景：

- 任何 p50 延迟比峰值吞吐量更重要的交互式对话负载
- 代码生成和结构化输出（JSON、SQL）——目标分布高度可预测，`α` 超过 0.9
- 长篇生成（数千词元）——摊销后的加速效果持续发挥

不适合的场景：

- 非常小的模型（< 3B）——草稿比验证器便宜的幅度有限
- 单请求 CPU 部署——草稿模型的内存开销可能得不偿失
- 高温度创意采样——`α` 会崩溃

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|---------|
| Leviathan 规则 | "min(1, q 除以 p)" | 以概率 `min(1, q(d)/p(d))` 进行 Bernoulli 接受/拒绝，拒绝时从残差采样即可精确保留验证器分布 |
| 残差分布 | "(q 减 p) 截零，归一化" | `(q - p)_+` 截断为零后归一化——拒绝时应采样的正确分布 |
| 接受率 α | "草稿正确的频率" | 拒绝规则下每词元的预期 Bernoulli 成功概率；决定所有加速比数学 |
| EAGLE-1 | "隐藏状态草稿" | 以验证器最后一层隐藏状态为条件的微型 Transformer 草稿（Li 等人，2024）|
| EAGLE-2 | "动态草稿树" | EAGLE-1 加上用树形注意力在一次验证器传播中打分的候选延续树 |
| EAGLE-3 | "训练时测试" | 去掉特征预测损失，在训练中以草稿自身输出为输入直接训练词元预测 |
| 训练时测试（TTT）| "曝光偏差修正" | 在训练中自回归运行草稿，使训练和测试输入分布匹配——是计划采样的直接类比 |
| KV 回滚 | "撤销被拒绝的草稿" | 在拒绝后将验证器 KV 缓存重置到被接受前缀长度的记账操作 |
| 奖励词元 | "那个免费的" | 当所有 `N` 个草稿都被接受时，无需额外验证器代价从 `q_{N+1}` 采样的额外词元 |
| 树形注意力 | "一次验证多个候选" | 使用尊重草稿树拓扑的非因果掩码的注意力；在一次前向传播中计算树中每个节点的 `q_i` |
