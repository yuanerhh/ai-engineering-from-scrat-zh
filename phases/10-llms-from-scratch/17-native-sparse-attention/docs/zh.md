# 原生稀疏注意力（DeepSeek NSA）

> 在 64k 词元时，注意力消耗了 70-80% 的解码延迟。每个开放模型实验室都有解决方案。DeepSeek 的 NSA（ACL 2025 最佳论文）是真正落地的那个：三个并行注意力分支——压缩粗粒度词元、选择性保留细粒度词元、滑动窗口用于局部上下文——通过学习的门控机制组合。它硬件对齐（内核友好）、原生可训练（在预训练中使用，而非推理时附加），在 64k 解码上运行速度快于 FlashAttention，同时匹配或超越完整注意力质量。本课端到端构建三个分支，并展示为何稀疏性是端到端可微的。

**类型：** 构建
**语言：** Python（标准库）
**前置条件：** Phase 7 · 第12课（KV 缓存，flash-attention），Phase 7 · 第15课（注意力变体），Phase 10 · 第16课（差分注意力）
**时间：** 约 60 分钟

## 学习目标

- 阐述 NSA 的三个注意力分支及各自捕捉的内容
- 解释为何 NSA 是"原生可训练"的，而此前的稀疏注意力方法只能用于推理
- 计算 NSA 相比完整注意力在 64k 上下文时的注意力计算节省，作为压缩块大小和选择 top-k 的函数
- 在标准库 Python 中对短合成序列实现三分支组合，并验证门控权重的行为

## 问题所在

序列长度 N 时，完整注意力的计算代价为 `O(N^2)`，每层 KV 缓存为 `O(N)`。在 64k 词元时，计算和内存带宽数字是灾难性的。NSA 论文中的理论估计：在 64k 时，注意力占据总解码延迟的 70-80%。所有下游指标——TTFT、tokens/秒、每百万词元成本——都由注意力代价主导。

稀疏注意力是显而易见的答案。此前的尝试分为两类：固定模式稀疏性（滑动窗口、步幅、块局部）丢弃信息，在长距离召回任务上失败；推理时稀疏性（KV 缓存剪枝、H2O、StreamingLLM）应用于在密集注意力上预训练的模型，只能恢复一部分潜在加速，因为模型从未被要求通过稀疏模式路由信息。

原生稀疏注意力（Yuan 等人，DeepSeek + 北大 + 华盛顿大学，ACL 2025 最佳论文，arXiv:2502.11089）两者兼顾：模型在预训练期间学习的稀疏模式，实现为真正在推理时兑现计算节省的内核对齐算法。两年后，NSA 或其直接后继者将成为每个前沿长上下文模型的默认注意力机制。

## 概念讲解

### 三个并行分支

对于每个查询，NSA 针对 KV 缓存的三种不同视图各运行一次注意力：

1. **压缩分支：** 词元被分组为大小 `l` 的块（通常为 32 或 64），每个块通过一个小型学习 MLP 压缩为单个摘要词元。查询在这些压缩词元上进行注意力，获得整个序列的粗粒度视图。

2. **选择分支：** 利用压缩分支的注意力分数，识别与当前查询最相关的 top-k 个块。从这些块中读取细粒度（未压缩）词元，查询在所有这些词元上进行注意力。可以将压缩分支注意力视为选择的路由信号。

3. **滑动窗口分支：** 查询在最近 `W` 个词元（通常为 512）上进行注意力，获取局部上下文。该分支捕捉其他两个分支可能遗漏的结构密集型短距离模式（语法、局部共指）。

三个分支输出通过学习的逐位置门控组合：

```
out = g_cmp * out_cmp + g_sel * out_sel + g_win * out_win
```

`g_cmp, g_sel, g_win` 是查询上一个小型 MLP 输出的门控权重，它们不必加和为 1——可以独立地对各分支加权。

### 为何"原生可训练"

选择步骤（top-k 块）是离散的。离散操作会打断梯度流。此前的稀疏注意力要么跳过通过选择的反向传播（限制训练），要么使用在推理时无法给出真实稀疏性的连续松弛。

NSA 绕开了这一问题：压缩分支注意力**本身就是**整个序列上可微的粗粒度注意力。top-k 操作只是重用压缩分支的顶部注意力分数来决定加载哪些细粒度块。梯度通过压缩分支分数流动（影响压缩输出和选择逻辑），所选块对最终输出的贡献也是可微的。不可微的 `top_k` 操作在前向计算图中是一个无操作——它只控制从内存加载哪些块。

这就是 NSA 能够端到端用于预训练的原因。模型学会联合通过三个分支路由信息，产生在推理时真正兑现加速承诺的稀疏模式。

### 硬件对齐内核

NSA 的内核专为现代 GPU 内存层次设计。内核按 GQA 组加载查询（外循环），为每组获取对应的稀疏 KV 块（内循环），并在 SRAM 中运行注意力。由于每个查询组看到相同的所选块（选择按查询组而非按查询头进行），KV 加载在组内摊销，算术强度保持较高。

论文报告 Triton 内核在 64k 解码上比 FlashAttention 快 9 倍，加速比随序列长度增长。前向和反向内核均已提供。

### 计算预算

设 `N` 为序列长度，`l` 为压缩块大小，`k` 为 top-k 选择数量，`w` 为滑动窗口，`b` 为所选块大小（通常等于 `l`）。

- 压缩分支：每查询 `O(N/l)` 个键，共 `O(N * N/l)`
- 选择分支：每查询 `O(k * b)` 个键，共 `O(N * k * b)`
- 滑动分支：每查询 `O(w)` 个键，共 `O(N * w)`

总计：`O(N * (N/l + k*b + w))`

当 `N = 64k, l = 64, k = 16, b = 64, w = 512` 时：每查询代价为 `1000 + 1024 + 512 = 2536` 个键。完整注意力为 64000 个键。**25 倍计算缩减。**

当 `N = 128k, l = 64, k = 16, b = 64, w = 512` 时：每查询代价为 `2000 + 1024 + 512 = 3536` 个键。完整注意力为 128000 个键。**36 倍缩减。** 收益随序列长度增长——这正是关键所在。

### 横向对比

| 方法 | 可微 | 真实推理加速 | 长距离召回 |
|-----|-----|------------|---------|
| 仅滑动窗口 | 是 | 是 | 失败 |
| 步幅/块稀疏 | 是 | 是 | 部分 |
| KV 剪枝（H2O、StreamingLLM）| N/A（推理时）| 是 | 部分 |
| MoBA（Moonshot）| 部分 | 是 | 良好 |
| NSA | 是（原生）| 是（64k 时 9x）| 匹配完整注意力 |

MoBA（Moonshot，arXiv:2502.13189）与 NSA 同期发表，采用类似的"三者优于一者"方法，将 MoE 原理应用于注意力块。NSA 和 MoBA 是 2026 年长上下文预训练需要了解的两种架构。

## 动手实践

`code/main.py` 在短合成序列上实现三个分支，展示：

- 压缩 MLP（为教学清晰度使用简单的均值池化基线；真实 NSA 使用学习的 MLP）
- 由压缩分支分数驱动的 top-k 块选择
- 最后 `w` 个词元上的滑动窗口注意力
- 门控组合
- 与完整注意力对比的计算量统计打印

### 第 1 步：将词元压缩成块

```python
def compress(K, l):
    n = len(K)
    n_blocks = (n + l - 1) // l
    out = []
    for b in range(n_blocks):
        start, end = b * l, min((b + 1) * l, n)
        block = K[start:end]
        summary = [sum(row[d] for row in block) / len(block) for d in range(len(K[0]))]
        out.append(summary)
    return out
```

### 第 2 步：压缩分支注意力

对压缩键运行查询的 softmax 注意力。压缩分支分数同时作为 top-k 选择的信号。

### 第 3 步：top-k 块选择

选取 `k` 个得分最高的压缩块的索引，从这些块加载原始未压缩词元，并对其运行注意力。

### 第 4 步：滑动窗口注意力

取最后 `w` 个词元，对其运行标准注意力。

### 第 5 步：门控 + 组合

查询上的小型 MLP 产生三个门控权重，最终输出为三个分支输出的加权求和。

### 第 6 步：计算量统计

打印每个分支每查询处理的键数以及总计，与 `N`（完整注意力）对比。在 1024 词元合成序列上，`l = 32, k = 4, w = 128` 时，NSA 每查询看到 `32 + 128 + 128 = 288` 个键，而完整注意力为 1024 个——少 3.5 倍。

## 实际运用

NSA 正在 DeepSeek 自己的长上下文预训练管线中使用。截至 2026 年 4 月，公共推理栈中的集成状态：

- **DeepSeek 内部**：原生支持，发布的权重使用 NSA 或其后继 DSA（Deepseek Sparse Attention）
- **vLLM**：针对 DeepSeek-V3.x 权重的实验性 NSA 支持正在开发中
- **SGLang**：NSA 基准测试已发布；生产路径跟随 vLLM
- **llama.cpp / CPU**：不支持；内核分解的开销在 CPU 吞吐量下不值得

何时适合使用 NSA：

- 以 64k+ 上下文为目标且具有足够计算预算的预训练或继续训练
- 推理 DeepSeek 自己的长上下文检查点——权重原生使用 NSA

何时不适合：

- 服务现有密集注意力预训练模型——不继续训练无法改造 NSA
- 上下文低于 16k——三分支开销超过节省的计算
- 对延迟敏感的批量为 1 的交互式对话——仅在长上下文时才有解码延迟收益

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|---------|
| 压缩分支 | "粗粒度视图" | 在块平均键上的注意力，每查询 O(N/l) 个键，提供全局上下文 |
| 选择分支 | "Top-k 块" | 在压缩分支得分最高的 `k` 个块的细粒度词元上进行注意力 |
| 滑动窗口 | "局部上下文" | 最近 `W` 个词元上的注意力，用于短距离模式 |
| 原生可训练性 | "预训练时开启稀疏性" | 稀疏模式在预训练期间学习，而非在推理时附加 |
| 压缩块大小 l | "粗粒度视图的组大小" | 合并为一个摘要的词元数量；通常为 32-64 |
| Top-k | "保留的块数" | 其未压缩词元被读取的压缩块数量；通常为 16 |
| 滑动窗口 W | "局部注意力半径" | 通常为 512；更短会损害局部连贯性，更长会浪费计算 |
| 分支门控 | "如何混合三者" | 对三个分支贡献加权的逐位置 MLP 输出 |
| 硬件对齐 | "内核友好的稀疏性" | 稀疏模式的选择使实际 GPU 内核能够实现理论加速 |
| DSA | "NSA 的后继" | Deepseek 稀疏注意力，NSA 在 DeepSeek 产品线中的后继架构 |
