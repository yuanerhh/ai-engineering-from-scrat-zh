# 操作预算、迭代上限和成本调节器

> 一家中型电商智能体的月度 LLM 成本在其团队启用"订单跟踪"技能后从 1,200 美元跳升至 4,800 美元。这不是定价漏洞。这是一个找到新循环并持续在其中消费的智能体。微软的 Agent 治理工具包（2026 年 4 月 2 日）将针对此类问题的防御措施编成了规范：每请求 `max_tokens`、每任务 token 和美元预算、每天/月上限、迭代上限、分级模型路由、提示词缓存、上下文窗口化、昂贵操作上的 HITL 检查点、预算突破时的终止开关。Anthropic 的 Claude Code Agent SDK 以不同的名称提供相同的基元。金融速度限制——例如在 10 分钟内消费超过 50 美元时切断访问——比月度上限更快捕获循环。

**类型：** 学习
**编程语言：** Python（标准库，分层成本调节器模拟器）
**前置知识：** Phase 15 · 10（权限模式）、Phase 15 · 12（持久执行）
**预计时间：** 约 60 分钟

## 问题背景

自主智能体每轮都花费真实的钱。聊天机器人的糟糕输出是一个糟糕的回复；智能体的糟糕循环是一张账单。该行业记录的故障模式术语是"钱包拒绝服务"——智能体持续推理、持续调用工具、持续计费，没有任何东西阻止它，因为没有任何东西被设计为阻止它。

修复方案不是一个数字。它是在不同时间尺度和粒度上的一系列限制：每请求、每任务、每小时、每天、每月。精心设计的堆栈在几分钟内捕获失控循环，在几小时内捕获慢速泄漏，在一天内捕获糟糕的发布。同样的堆栈在智能体是长时间且自主时维持预算。

这是一个工程课：数学是微不足道的，纪律是团队失败的地方。下面的限制列表都在微软 Agent 治理工具包或 Anthropic Claude Code Agent SDK 文档中有名称。

## 核心概念

### 成本调节器堆栈

1. **每请求 `max_tokens`。** 简单。防止任何一次调用发出无界完成。
2. **每任务 token 预算。** 在整个运行中，不超过 N 个 token。在上限处硬停。
3. **每任务美元预算。** 与 token 相同但以货币计。Claude Code 中的 `max_budget_usd`。
4. **每工具调用上限。** 不超过 N 次 `WebFetch` 调用、N 次 `shell_exec` 调用等。
5. **迭代上限（`max_turns`）。** 总智能体循环迭代次数；防止无限推理循环。
6. **每分钟/每小时/每天/每月上限。** 滚动窗口。在不同时间尺度捕获泄漏。
7. **金融速度限制。** 例如，"如果消费在 10 分钟内超过 50 美元，切断访问"。在月度上限触发之前捕获基于循环的消耗。
8. **分级模型路由。** 默认使用较小的模型；仅当分类器判断任务需要时才升级到较大的模型。
9. **提示词缓存。** 系统提示和稳定上下文存储在提供商缓存中；重新发送的 token 成本接近零。
10. **上下文窗口化。** 压缩/摘要以将活跃上下文保持在阈值以下；直接降低 token 成本。
11. **昂贵操作上的 HITL 检查点。** 在已知昂贵的操作（长工具调用、大下载、代价高昂的模型升级）之前，需要人工确认。
12. **预算突破时的终止开关。** 任何上限触发时会话中止。上限被记录；需要单独的重新启用路径。

### 为什么需要堆栈而不是一个上限

单个月度上限只有在钱包耗尽后才能捕获失控智能体。单个每请求上限在会话级别什么都捕获不到。不同的故障模式需要不同的时间尺度：

- **失控循环**（智能体在 5 秒重试中卡住）：由速度限制捕获。
- **慢速泄漏**（智能体每任务做约 2 倍预期工作）：由每日上限捕获。
- **糟糕的发布**（新版本使用 5 倍 token）：由每周/月度上限捕获。
- **合法激增**（真实需求，不是漏洞）：由小时/天上限捕获，带清晰日志。

### Claude Code 的预算面

Claude Code Agent SDK 提供（公开文档）：

- `max_turns` — 迭代上限。
- `max_budget_usd` — 美元上限；突破时会话中止。
- `allowed_tools` / `disallowed_tools` — 工具允许列表和拒绝列表。
- 工具使用前的钩子点用于自定义成本核算。

与权限模式阶梯结合（第 10 课）。没有 `max_budget_usd` 的 `autoMode` 会话是未受约束的自主性。Anthropic 明确将自动模式框架为需要预算控制；分类器与成本无关。

### EU AI 法案、OWASP Agentic Top 10

微软的 Agent 治理工具包涵盖了 OWASP Agentic Top 10 和 EU AI 法案第 14 条（人类监督）要求。对于 EU 的生产部署，日志记录和上限执行不是可选的。

### 观察到的 1,200 美元 → 4,800 美元案例

微软文档中的真实案例：一个电商智能体在添加新工具后月度成本翻了三倍。该工具允许智能体在每个会话期间轮询订单状态。没有循环检测。没有每工具上限。没有逐周增长警报。修复方案是每工具上限加上每日增长警报。这是一个模板：每个新工具面都是一个新的潜在循环；每个新工具都需要自己的上限和自己的警报。

## 动手实践

`code/main.py` 模拟有和没有分层成本调节器堆栈的智能体运行。模拟智能体在几轮后漂入轮询循环；分层堆栈在速度窗口内捕获它，而单个月度上限要到几天后才会触发。

## 产出技能

`outputs/skill-agent-budget-audit.md` 审计提议的智能体部署的成本调节器堆栈，并标记缺失的层。

## 练习

1. 运行 `code/main.py`。确认速度限制在轮询循环轨迹上在迭代上限之前触发。现在禁用速度限制，测量智能体在迭代上限捕获它之前"消费"了多少。

2. 为浏览器智能体（第 11 课）设计一个每工具上限集。哪个工具需要最严格的上限？哪个工具可以无限制运行而无风险？

3. 阅读微软 Agent 治理工具包文档。列出工具包命名的每种上限类型。将每种映射到一个故障模式（失控循环、慢速泄漏、糟糕发布、激增）。

4. 为一个现实任务（例如，"在仓库中分类 50 个 Issue"）定价一次隔夜无人值守运行。将 `max_budget_usd` 设置为你的点估计的 2 倍。为这个 2 倍辩护。

5. Claude Code 的 `max_budget_usd` 在会话聚合成本上触发。设计一个你会外部执行的补充速度限制。什么触发截断，重新启用是什么样子的？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 钱包拒绝服务 | "失控账单" | 智能体循环产生消费，没有上限阻止它 |
| max_tokens | "每请求上限" | 单个完成大小的上限 |
| max_turns | "迭代上限" | 会话中智能体循环迭代次数的上限 |
| max_budget_usd | "美元终止开关" | 会话成本上限；突破时中止 |
| 速度限制 | "速率上限" | 每短窗口消费限制（例如，10 分钟 50 美元） |
| 分级路由 | "小模型优先" | 廉价模型默认；仅当分类器认为值得时才升级 |
| 提示词缓存 | "缓存的系统提示" | 提供商端缓存将重新发送 token 成本降至接近零 |
| HITL 检查点 | "人工审批门" | 昂贵操作前需要人工确认 |
