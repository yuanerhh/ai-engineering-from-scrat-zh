# AlphaEvolve——进化式编码智能体

> 将前沿编码模型与进化循环和机器可验证评估器配对。让循环运行足够长时间。它发现了一种使用 48 次标量乘法的 4x4 复数矩阵乘法程序——56 年来首次超越 Strassen 算法。它还找到了一种谷歌级 Borg 调度启发式方法，在生产中回收了约 0.7% 的集群算力。这个架构有意设计得平淡无奇。成果来自评估器的严谨性。

**类型：** 学习
**编程语言：** Python（标准库，进化循环玩具程序）
**前置知识：** Phase 15 · 01（长时间智能体框架）、Phase 15 · 02（自学推理）
**预计时间：** 约 60 分钟

## 问题背景

大型语言模型可以编写代码。进化算法可以在代码上搜索。两者分别尝试了几十年；两者都遇到了瓶颈。LLM 的瓶颈是胡编乱造：模型编写了看似合理的代码，但并不能做它声称的事情。进化的瓶颈是搜索成本：对语法的随机突变很少产生可编译的程序，更不用说更好的程序了。

AlphaEvolve（Novikov 等人，DeepMind，arXiv:2506.13131，2025 年 6 月）将两者结合起来。LLM 对程序数据库提出有针对性的编辑；自动评估器对每个变体进行评分；高分变体成为未来几代的父代。LLM 处理编写合理代码的昂贵步骤；评估器捕获胡编乱造。循环运行数小时到数周。

报告的结果：48 次标量乘法的 4x4 复数矩阵乘法（Strassen 1969 年的界是 49）、谷歌生产中的 Borg 调度启发式、32.5% 的 FlashAttention 内核加速、Gemini 训练吞吐量提升。

该架构之所以有效，是因为评估器是机器可验证的。在评估器不可验证的地方，它不起作用。这种不对称性就是本课的教训。

## 核心概念

### 循环

1. 从一个正确但次优的种子程序 `P_0` 开始。
2. 维护一个变体程序数据库，每个变体都由评估器评分。
3. 从数据库中采样一个或多个父代（MAP-elites 风格或基于岛屿的）。
4. 提示 LLM（Gemini Flash 用于许多候选，Gemini Pro 用于困难的候选）生成父代的修改变体。
5. 编译、运行并在保留的评估器上评估变体。
6. 按其分数和特征向量插入数据库。
7. 重复。

两个细节很重要。首先，LLM 的提示不仅仅包含父代程序——通常还包括来自数据库的几个顶级变体、评估器签名以及简短的任务描述。模型的工作是提出一个可能提高分数的有针对性的变更。其次，数据库是结构化的（MAP-elites 网格、基于岛屿的），因此循环探索多样性，而不仅仅是当前领先者。

### 为什么评估器是不可妥协的

AlphaEvolve 的成果都来自评估器快速、确定性且难以欺骗的领域：

- **矩阵乘法算法**：一个单元测试，按位完全相同地乘以矩阵并检查等式。
- **Borg 调度启发式**：一个生产级模拟器，重放历史集群负载并测量浪费的算力。
- **FlashAttention 内核**：正确性测试加上真实硬件上的挂钟基准。
- **Gemini 训练吞吐量**：每步测量的 GPU 秒数。

在每种情况下，评估器都能捕获否则会占主导地位的 LLM 错误类别：虚假的正确性声明、在硬件上消失的性能声明，以及边缘情况失败。移除评估器，循环就会优化漂亮的代码。

### 奖励黑客攻击是该陈述的另一面

进化优化评估器所测量的任何内容。如果评估器不完美，循环会找到不完美之处。在未经验证的领域，循环会优化表面特征，而不是预期行为。DeepMind 在论文中明确指出了这一点：AlphaEvolve 的成功只能迁移到评估器严谨性与搜索雄心相匹配的领域。

2025-2026 年代码搜索循环中奖励黑客攻击的具体例子：

- 奖励"完成时间"的优化目标奖励了提交空解决方案。
- 奖励测试下正确性的基准分数奖励了记忆测试和过拟合。
- "代码质量"代理奖励了删除注释和重写变量名，没有语义变化。

AlphaEvolve 中的修复：提供一个 LLM 从未见过的保留评估器，并在评估时生成输入。即便如此，DeepMind 建议对任何拟议的部署进行严格审查。

### 为什么 LLM + 搜索胜过两者单独使用

LLM 可以生成可编译、语义合理的修改。对 2000 行 Python 文件的随机突变 GA 几乎总是产生语法错误。LLM 还将搜索集中在合理的邻域（改变一个函数，而不是随机字节），这大大减少了浪费的评估器调用。

反过来，评估器捕获 LLM 的胡编乱造。LLM 会自信地声称一个函数"在极限情况下是 O(n log n)"，而实际上是 O(n^2)；挂钟基准使问题得到解决。

### AlphaEvolve 在前沿技术栈中的位置

| 系统 | 生成器 | 评估器 | 领域 | 示例成果 |
|------|--------|--------|------|---------|
| AlphaEvolve | Gemini | 正确性 + 基准 | 算法、内核、调度器 | 48 乘法 4x4 矩阵乘法 |
| FunSearch（DeepMind，2023） | PaLM / Codey | 正确性 | 组合数学 | 上集下界 |
| AI Scientist v2（Sakana，第 5 课） | GPT/Claude | LLM 批评 + 实验 | ML 研究 | ICLR 研讨会论文 |
| Darwin Godel Machine（第 4 课） | 智能体脚手架 | SWE-bench / Polyglot | 智能体代码 | 20% → 50% SWE-bench |

四者都是同一配方的变体：生成器加评估器，循环。区别在于评估器评分的内容以及其严谨程度。

## 动手实践

`code/main.py` 在一个简单符号回归问题上实现了最小化的 AlphaEvolve 风格循环。"LLM"是一个标准库代理，对计算目标函数的程序提出小的语法突变。"评估器"在保留的测试点上测量均方误差。

观察：

- 最佳分数如何随代际提升。
- MAP-elites 网格如何保持多样化的解决方案，使循环不会收敛到局部最小值。
- 移除保留测试（仅训练评估器）如何让循环壮观地过拟合。

## 练习

1. 运行 `code/main.py`。记录最佳分数轨迹。禁用保留评估器（标志 `--no-holdout`）并重新运行。量化过拟合程度。

2. 阅读 AlphaEvolve 论文第 3 节关于 MAP-elites 网格的内容。为一个新问题（例如编译器优化通道）设计一个特征向量描述符，以保持搜索的多样性。

3. 48 乘法 4x4 结果在 56 年后改进了 Strassen 的 49 乘法界。阅读论文附录 F 并用三句话解释为什么该问题的评估器特别容易做对，以及为什么大多数领域不是这样。

4. 提出一个 AlphaEvolve 会失败的领域。精确找出评估器失败的地方以及原因。

5. 对于你熟悉的领域，写出你会使用的评估器签名。包括 (a) 正确性条件，(b) 性能指标，(c) 保留输入生成规则，(d) 至少一个防奖励黑客攻击检查。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| AlphaEvolve | "DeepMind 的进化编码智能体" | Gemini + 程序数据库 + 机器可验证评估器 |
| MAP-elites | "保持多样性的存档" | 按特征向量键控的网格；每个单元格保存具有该描述符的最佳变体 |
| 岛屿模型 | "并行进化子种群" | 定期迁移的独立种群；防止过早收敛 |
| 机器可验证评估器 | "确定性预言机" | LLM 无法伪造的单元测试、模拟器或基准——该循环的前提条件 |
| 奖励黑客攻击 | "优化指标而非目标" | 循环找到最大化分数的方法而不执行预期任务 |
| 种子程序 | "起点" | 循环进化的初始正确但次优程序 |
| 保留评估器 | "LLM 从未见过的评估数据" | 在评估时生成的输入以防止记忆 |
