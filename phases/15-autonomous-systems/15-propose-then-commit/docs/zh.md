# 人机协作：提议-然后-提交

> 2026 年关于 HITL 的共识是具体的。它不是"智能体询问，用户点击批准"。它是提议-然后-提交：提议的操作持久化到带有幂等键的持久存储；向审查者展示意图、数据溯源、触及的权限、爆炸半径和回滚计划；仅在积极确认后提交；执行后验证以确认副作用实际发生了。LangGraph 的 `interrupt()` 加 PostgreSQL 检查点、微软 Agent Framework 的 `RequestInfoEvent` 和 Cloudflare 的 `waitForApproval()` 都实现了相同的形状。典型故障模式是橡皮图章审批："批准？"在没有审查的情况下被点击。已记录的缓解措施是带明确检查清单的挑战-响应机制。

**类型：** 学习
**编程语言：** Python（标准库，带幂等性的提议-然后-提交状态机）
**前置知识：** Phase 15 · 12（持久执行）、Phase 15 · 14（触发线）
**预计时间：** 约 60 分钟

## 问题背景

智能体采取一个操作。用户必须决定：批准还是不批准。如果决定是即时的，它可能不是一个审查。如果决定是结构化的，它是缓慢但可信的。工程问题是如何使结构化审查成为阻力最小的路径。

2023 年的 HITL 模式是一个同步提示："智能体想要向 X 发送主体为 Y 的电子邮件——批准？"用户点击批准。每个人都觉得系统是安全的。实际上，这个面受到严重的橡皮图章化：用户快速批准，批准预测性很差，当智能体出错时，审计追踪显示用户无法回忆的长批准历史。

2026 年的模式——提议-然后-提交——将 HITL 移到持久基底上，附加结构化元数据，并要求积极提交。每个托管智能体 SDK 都提供一个版本：LangGraph `interrupt()`、微软 Agent Framework `RequestInfoEvent`、Cloudflare `waitForApproval()`。API 名称不同；形状不变。

## 核心概念

### 提议-然后-提交状态机

1. **提议。** 智能体生成提议的操作。持久化到持久存储（PostgreSQL、Redis、Durable Object）。包括：
   - 意图（智能体为什么这样做）
   - 数据溯源（什么来源导致了这个提议）
   - 触及的权限（哪些范围/文件/端点）
   - 爆炸半径（最坏情况是什么）
   - 回滚计划（如果提交了，如何撤销）
   - 幂等键（每个提议唯一；重新提交返回相同记录）
2. **展示。** 审查者看到带所有元数据的提议。审查者是人（不是智能体审查自身）。
3. **提交。** 积极确认。操作执行。
4. **验证。** 执行后，读回副作用并确认。如果验证步骤失败，系统处于已知的错误状态，警报响起。

### 幂等键

没有幂等键，在瞬时失败后的重试可以双重执行批准的操作。具体例子：用户批准"从 A 向 B 转账 100 美元"。网络抖动。工作流重试。用户批准了一次，但转账执行了两次。幂等键将审批绑定到单一的、唯一的副作用；第二次执行是空操作。

这与 Stripe 和 AWS API 使用的相同幂等模式。将其重用于智能体批准在微软 Agent Framework 文档中是明确的。

### 持久性：为什么审批比进程更持久

等待审批的区域是智能体不拥有的状态片段。工作流被暂停（第 12 课）。当审批到来时，工作流从该点恢复。这就是为什么 LangGraph 将 `interrupt()` 与 PostgreSQL 检查点配对而不仅仅是内存中状态——两天后的审批仍然能找到完整的工作流。

### 橡皮图章审批和挑战-响应缓解

HITL 的默认 UI（"批准"/"拒绝"按钮）产生没有真正审查的快速审批。已记录的缓解措施：在启用批准按钮之前需要对特定问题给出肯定答案的挑战-响应检查清单。具体形状：

- "你理解这触及什么资源吗？[ ]"
- "你已验证爆炸半径是可接受的吗？[ ]"
- "如果这失败了，你有回滚计划吗？[ ]"

不是为了官僚主义而设置的——是一个强制函数。无法勾选框的审查者要么请求澄清（升级），要么拒绝（安全默认）。Anthropic 的智能体安全研究明确引用检查清单驱动的 HITL 作为橡皮图章审批模式的缓解措施。

### 什么算作后果性

并非每个操作都需要提议-然后-提交。2026 年的指导：

- **后果性操作**（始终 HITL）：不可逆写入、金融交易、出站通信、生产数据库更改、破坏性文件系统操作。
- **可逆操作**（有时 HITL）：本地文件编辑、暂存环境更改、带明确回滚的可逆写入。
- **读取和检查**（从不 HITL）：读取文件、列出资源、调用只读 API。

### 操作后验证

"提交运行了"与"副作用发生了"不同。网络分区和竞争条件可以产生一个认为成功但后端没有持久化的工作流。验证步骤在提交后重新读取目标资源以确认。这与带 `RETURNING` 子句的数据库事务或 `PutObject` 后的 AWS `GetObject` 是相同的模式。

### EU AI 法案第 14 条

第 14 条规定对欧盟高风险 AI 系统进行有效的人类监督。"有效"不是装饰性的。监管语言明确排除橡皮图章模式。带挑战-响应的提议-然后-提交是微软 Agent 治理工具包合规文档中能通过第 14 条审查的形状。

## 动手实践

`code/main.py` 在标准库 Python 中实现了提议-然后-提交状态机。持久存储是一个 JSON 文件。幂等键是（thread_id、action_signature）的哈希。驱动程序模拟三种情况：清洁审批流程、瞬时失败后的重试（不得双重执行），以及橡皮图章默认与挑战-响应流程的对比。

## 产出技能

`outputs/skill-hitl-design.md` 审查提议的 HITL 工作流的提议-然后-提交形状，并标记缺少的元数据、幂等性、验证或挑战-响应层。

## 练习

1. 运行 `code/main.py`。确认批准提议的重试使用持久记录而不重新执行。现在将幂等键改为包含时间戳，显示重试双重执行。

2. 用 `rollback` 字段扩展提议记录。模拟验证步骤失败的执行。显示回滚自动触发。

3. 阅读微软 Agent Framework 的 `RequestInfoEvent` 文档。找出 API 包含的一个玩具引擎缺少的元数据字段。添加它并解释它防范什么。

4. 为特定操作（例如，"发布到公共 Twitter 账户"）设计挑战-响应检查清单。审查者必须回答哪三个问题？为什么是那三个？

5. 选择一个同步"批准？"提示就足够的情况（不需要持久存储）。解释原因，并命名你接受的风险类别。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 提议-然后-提交 | "两阶段审批" | 持久化提议 + 积极提交 + 验证 |
| 幂等键 | "重试安全令牌" | 每个提议唯一；第二次执行为空操作 |
| 数据溯源 | "它来自哪里" | 导致提议的具体来源内容 |
| 爆炸半径 | "最坏情况" | 如果操作出错的影响范围 |
| 橡皮图章 | "快速审批" | 在没有真正审查的情况下点击"批准" |
| 挑战-响应 | "强制检查清单" | 审查者必须积极确认特定问题 |
| RequestInfoEvent | "MS Agent Framework 基元" | 带结构化元数据的持久 HITL 请求 |
| `interrupt()` / `waitForApproval()` | "框架基元" | 相同形状的 LangGraph / Cloudflare 等价物 |
