# Claude Code 作为自主智能体：权限模式与自动模式

> Claude Code 提供七种权限模式。"plan"在每次操作前询问，"default"只在危险操作时询问，"acceptEdits"自动批准文件写入但仍确认 shell 执行，"bypassPermissions"批准一切。自动模式（2026 年 3 月 24 日）用两阶段并行安全分类器取代了逐操作审批：单 token 快速检查在每次操作上运行；标记的操作触发思维链深度审查。操作预算通过 `max_turns` 和 `max_budget_usd` 强制执行。自动模式作为研究预览发布——Anthropic 明确表示分类器本身并不充分。

**类型：** 学习
**编程语言：** Python（标准库，两阶段分类器模拟器）
**前置知识：** Phase 15 · 01（长时间智能体）、Phase 15 · 09（编码智能体全景）
**预计时间：** 约 45 分钟

## 问题背景

机器上的自主编码智能体是一个独特的安全类别。攻击面是智能体可以到达的一切——文件系统、网络、凭据、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier 等人公开指出了这一点：计算机使用智能体不是聊天机器人的"功能更新"，它们是具有全新风险配置的新型工具。

Claude Code 的权限系统是 Anthropic 的答案。不是一个"自主/非自主"开关，而是七种模式，跨越一个能力阶梯：plan → default → acceptEdits → … → bypassPermissions。每种模式是速度与每操作审查之间的不同权衡。自动模式（2026 年 3 月）添加了一个两阶段分类器，对分类器判断为安全的操作将审批从用户的关键路径上移走，同时为分类器标记的操作保留审查层。

工程问题：该系统捕获什么，错过什么，以及给定任务实际上需要哪种模式？

## 核心概念

### 七种权限模式

| 模式 | 行为 | 何时使用 |
|------|------|---------|
| `plan` | 智能体提出计划；用户批准整个计划；每次操作在执行前都被审查 | 不熟悉的任务；生产相邻代码；第一次在仓库上使用智能体 |
| `default` | 智能体执行操作；对任何"危险"操作（shell 执行、破坏性操作、网络调用）提示用户 | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；shell 执行和网络调用仍然提示 | 跨多个文件的重构过程 |
| `acceptExec` | shell 命令在策划的允许列表内自动批准；写入自动批准 | 每个 shell 命令都是 `npm test` 之类的紧密内循环 |
| `autoMode` | 两阶段安全分类器；标记的操作升级为审查 | 受限工作区中的长时间无人值守运行 |
| `yolo` | 跳过大多数提示；仍然运行工具允许列表/拒绝列表 | 临时沙盒、CI 作业、研究脚本 |
| `bypassPermissions` | 批准一切 | 记录为"仅在您愿意丢弃的临时容器内" |

（上面的名称与公开的 Claude Code 文档匹配；"yolo"是许可中间地带的非正式简称。）

### 自动模式，一页概括

自动模式（2026 年 3 月 24 日推出）是第一个将每操作审批委托给模型的权限模式。结构：

1. **阶段 1——单 token 分类器。** 在每个提议的操作上运行。廉价。返回"安全"或"标记"。设计为与主智能体循环并行运行，因此批准的操作不会在队列中等待。
2. **阶段 2——次级策略/安全审查。** 仅在标记的操作上运行。对操作、会话的当前状态和声明的任务执行专注的策略和安全评估。如果置信度低，则升级给用户 HITL。

预算控制与分类器并排：

- `max_turns` — 会话中的总迭代次数。
- `max_budget_usd` — 中止会话的美元上限。
- 每个工具的操作计数限制（不超过 N 次 `WebFetch` 调用等）。

### 系统捕获什么

- 工具输入中直接的提示词注入，其中注入的指令映射到已知危险的操作形状。
- 重复工具循环——分类器可以看到操作 N+1 与操作 N 几乎相同，连续五次。
- 在其他仅文件编辑会话上明显超出范围的 shell 命令。

### 系统可能错过什么

- **微妙的提示词注入**，在不产生单个标记操作的情况下调制行为。间接提示词注入不是完全可修补的漏洞（OpenAI 准备主管，2025 年，关于浏览器智能体——见第 11 课）。
- **语义级别的错误行为。** 每个单独的操作看起来可能是安全的，而组合轨迹是有害的。分类器判断操作；它不重新推导用户的意图。
- **通过合法渠道泄露。** 将数据写入您拥有的文件，然后 `git push` 到公共仓库，是一系列允许操作，其组合就是问题所在。

### 研究预览框架

Anthropic 将自动模式作为研究预览发布。文档明确表示分类器是一个层，而不是解决方案：用户被期望将自动模式与预算、允许列表、隔离工作区和轨迹审计（第 12-16 课）结合使用。预览框架也反映了已记录的评估与部署差距（第 1 课）——通过离线评估的分类器在用户上下文模糊的真实会话中可能行为不同。

### 这个阶梯在您工作流程中的位置

- 不熟悉的任务：从 `plan` 开始。阅读计划比回滚糟糕的运行更便宜。
- 已知的重构：`acceptEdits` 节省大量确认点击。
- 无人值守的后台运行：`autoMode` 仅在您已测量爆炸半径的工作区内（无凭据、无生产挂载、无您未选择的出口）。
- 临时容器：`yolo` / `bypassPermissions` 当且仅当容器及其凭据是可丢弃的时才可接受。

## 动手实践

`code/main.py` 模拟两阶段分类器。阶段 1 是提议操作上的廉价关键词规则；阶段 2 是较慢的多规则审查器。驱动程序馈入一个短的合成轨迹（安全操作、提示词注入尝试、重复循环），并显示分类器在哪里捕获，在哪里错过。

## 产出技能

`outputs/skill-permission-mode-picker.md` 将任务描述与正确的权限模式、预算上限和所需隔离匹配。

## 练习

1. 运行 `code/main.py`。哪种合成操作类型从不被阶段 1 标记，但总是被阶段 2 捕获？哪种两者都捕获不到？

2. 扩展阶段 1 规则集以捕获特定的已知危险形状（例如 `curl $ATTACKER/exfil`）。测量良性操作样本上的误报率。

3. 阅读 Anthropic 的"智能体循环如何工作"文档。列出智能体在 `default` 模式下默认触及的每个外部状态。在无人值守地运行 `autoMode` 之前，您需要单独门控哪些？

4. 设计一个 24 小时无人值守运行预算：`max_turns`、`max_budget_usd`、每工具上限、允许列表。为每个数字辩护。

5. 描述一个轨迹，其中每个单独的操作都被阶段 1 和阶段 2 批准，但组合行为是错位的。（第 14 课介绍了终止开关和金丝雀令牌如何解决这个问题。）

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 权限模式 | "智能体能做多少" | 七种控制每操作审批的命名策略之一 |
| plan 模式 | "什么都问" | 智能体编写计划；用户在执行前批准 |
| acceptEdits | "让它写文件" | 文件写入自动批准；shell 执行仍然提示 |
| autoMode | "自动审批" | 两阶段安全分类器；标记的操作升级 |
| bypassPermissions | "完全 YOLO" | 批准一切；用于临时容器 |
| 阶段 1 分类器 | "快速 token 检查" | 对提议操作的单 token 规则；并行运行 |
| 阶段 2 分类器 | "深度审查" | 对标记操作的思维链推理 |
| 研究预览 | "非正式发布" | Anthropic 对故障模式仍在映射中的功能的框架 |
