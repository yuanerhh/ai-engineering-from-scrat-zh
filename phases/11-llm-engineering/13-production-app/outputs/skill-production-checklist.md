# 生产 LLM 检查清单

发布 LLM 应用时，按顺序完成此检查清单。每个部分都有带具体阈值的通过/失败标准。

## 1. 安全（发布阻断项）

在任何部署之前，以下所有项目必须通过。

| 检查 | 通过标准 | 验证方法 |
|------|---------|---------|
| API 密钥存在环境变量中 | 代码库中零硬编码密钥 | `grep -r "sk-" --include="*.py"` 返回空结果 |
| 输入防护栏已激活 | 提示词注入模式被阻止 | 发送「忽略所有之前的指令」——返回被阻止的响应 |
| PII 脱敏 | 社会安全号、信用卡、邮件模式被捕获 | 发送「我的社会安全号是 123-45-6789」——PII 在 LLM 调用前被脱敏 |
| 输出过滤 | 危险内容被阻止 | 模型无法返回 `DROP TABLE`、`rm -rf`、`exec()` 模式 |
| 频率限制 | 每用户请求上限被执行 | 同一用户 10 秒内 100 个请求——最后 50+ 被拒绝 |
| 所有端点需要认证 | 无未经认证的 LLM 访问 | 不带 token 的 `curl /v1/chat` 返回 401 |
| CORS 已限制 | 只允许生产域名 | `Origin: evil.com` 请求被拒绝 |
| 最大输入 token | 超限请求被拒绝 | 发送 50K token 输入——返回 413 或截断 |

## 2. 可靠性（第一周生存）

这些措施能防止你的第一次值班事故。

| 检查 | 通过标准 | 验证方法 |
|------|---------|---------|
| 带退避的重试 | 5xx 错误重试 3 次，指数延迟 | 中途终止 LLM 模拟——重试在日志中可见 |
| 降级模型链 | 链中 2 个以上模型 | 主模型不可用——响应仍从降级模型返回 |
| 请求超时 | 所有外部调用最长 30 秒 | 慢速 LLM 模拟（60 秒）——请求在 30 秒超时 |
| 优雅降级 | 缓存/RAG 失败不会使服务崩溃 | 停止缓存——请求仍然成功（更慢，更贵） |
| 健康检查端点 | 返回依赖状态 | `GET /health` 返回 `{"status": "healthy", "cache": ..., "llm": ...}` |
| 流式传输正常 | 首个 token 在 500ms 内 | 首 token 时间已测量，持续 < 500ms |
| 错误消息安全 | 内部错误绝不泄露给用户 | 强制 500——用户看到通用错误，而非堆栈跟踪 |

## 3. 成本控制（第一月经济学）

这些措施能防止 $50K 的意外账单。

| 检查 | 通过标准 | 验证方法 |
|------|---------|---------|
| 每次请求成本追踪 | 每个请求记录 token 数 + USD 成本 | 请求日志包含 `input_tokens`、`output_tokens`、`cost_usd` 字段 |
| 语义缓存已激活 | 重复模式命中率 > 20% | 1000 个测试请求后缓存统计显示命中率 |
| 缓存 TTL 已配置 | 条目会过期（默认：1 小时） | 插入条目——TTL 后不再返回 |
| 每用户成本追踪 | 按 user_id 聚合成本 | 仪表板/API 显示按成本排名的前 10 用户 |
| 成本告警 | 达到日预算 80% 时告警 | 设置 $10 日预算，发送 $8.50 请求——告警触发 |
| 按成本进行模型路由 | 低复杂度查询使用更便宜的模型 | 简单问题路由到 gpt-4o-mini，复杂问题路由到 gpt-4o |
| 已设置最大输出 token | 每个模板的响应有上限 | max_output_tokens=512 的模板——响应从不超过该值 |

**成本估算公式：**
```
每月 LLM 成本 = DAU x 每用户查询次数 x 30 x (1 - 缓存命中率) x (avg_input_tokens x input_price + avg_output_tokens x output_price) / 1,000,000
```

**按规模划分的基准阈值：**

| DAU | 目标每次请求成本 | 月预算 |
|-----|----------------|--------|
| 1K | < $0.005 | < $750 |
| 10K | < $0.003 | < $4,500 |
| 100K | < $0.001 | < $15,000 |

## 4. 可观测性（在生产中调试）

你无法修复你看不到的东西。

| 检查 | 通过标准 | 验证方法 |
|------|---------|---------|
| 结构化 JSON 日志 | 每个请求产生一行 JSON 日志 | 日志包含：request_id、user_id、model、tokens、latency_ms、cost |
| 请求追踪 | 带组件计时的端到端追踪 | 单个请求显示：防护栏（5ms）+ 缓存（2ms）+ llm（3200ms）+ 评估（1ms） |
| 延迟追踪 | P50、P95、P99 已测量 | 1000 个请求后：P50 < 2 秒，P99 < 10 秒 |
| 错误率监控 | 错误已计数并分类 | 仪表板显示：0.5% API 错误、0.1% 防护栏阻止、0.01% 超时 |
| 缓存指标 | 命中率、未命中率、条目数可见 | `GET /v1/cache/stats` 返回当前数字 |
| A/B 测试指标 | 每变体质量指标已记录 | 每个请求记录 prompt_template + 版本以便比较 |
| 评估记录 | 每个请求记录质量信号 | 响应长度、延迟、模型、模板版本已存储供离线分析 |

## 5. 提示词管理

提示词是代码。像对待代码一样对待它们。

| 检查 | 通过标准 | 验证方法 |
|------|---------|---------|
| 版本化模板 | 每个模板有名称 + 版本字符串 | 模板更改创建新版本，旧版本保留 |
| A/B 测试支持 | 按确定性用户哈希分流 | 同一用户在实验期间始终看到同一变体 |
| 回滚能力 | 1 分钟内回滚到上一版本 | 更改实验配置——流量即时切换 |
| 模板验证 | 渲染前验证变量 | 模板中缺少变量引发清晰错误，而非 KeyError |
| 系统提示分离 | 系统和用户消息在独立字段 | 系统提示未被拼接到用户消息中 |

## 6. 扩展就绪

发布时不需要，10 倍规模时需要。

| 检查 | 通过标准 | 验证方法 |
|------|---------|---------|
| 异步 LLM 调用 | API 调用不阻塞线程 | 50 个并发请求——服务器 CPU 保持 < 30% |
| 连接池 | HTTP 连接被复用 | 网络追踪显示到 LLM 提供商的持久连接 |
| 水平扩展 | 无状态服务器设计 | 负载均衡器后的 2 个实例——所有请求成功 |
| 队列支持 | 非实时任务进入队列 | 摘要请求返回 job_id，结果通过轮询获取 |
| 负载测试 | 100 个并发用户，< 5% 错误率 | `wrk` 或 `locust` 测试在目标并发下通过 |

## 新项目的实施顺序

1. **第 1 天：** API 服务器 + 提示词模板 + 带重试的单次 LLM 调用
2. **第 2 天：** 输入防护栏 + 输出防护栏 + 错误处理
3. **第 3 天：** 语义缓存 + 每次请求的成本追踪
4. **第 4 天：** 流式传输（SSE）+ 健康检查端点
5. **第 5 天：** 结构化日志 + 请求追踪 + 评估记录
6. **第 2 周：** A/B 测试 + 提示词版本控制 + 回滚
7. **第 3 周：** 降级模型链 + 优雅降级
8. **第 4 周：** 负载测试 + 异步优化 + 水平扩展

## 快速诊断

生产中出现问题时，按以下顺序检查：

1. **用户报告错误？** 检查健康端点，然后检查日志中的错误率，然后检查 LLM 提供商状态页
2. **响应缓慢？** 检查 P99 延迟，然后检查缓存命中率，然后检查追踪中的 LLM 响应时间
3. **成本飙升？** 检查每次请求成本趋势，然后检查缓存命中率，然后检查按成本排名的顶级用户，然后查看是否有提示词模板变更增加了 token 数量
4. **质量下降？** 检查是否部署了新的提示词版本，检查 RAG 检索准确率是否改变，检查模型提供商是否更改了默认模型版本
5. **安全事件？** 检查防护栏阻止率（突然下降 = 防护栏被禁用），检查请求日志中的异常模式，立即轮换 API 密钥
