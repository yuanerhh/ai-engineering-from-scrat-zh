# 编排模式：监督者、蜂群、层次化

> 2026 年框架中反复出现四种编排模式：监督者-工作者、蜂群/点对点、层次化、辩论。Anthropic 的指导原则："关键在于为你的需求构建正确的系统。"从简单开始；只有当单个智能体加五种工作流模式不够时，才添加拓扑。

**类型：** 学习 + 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 12（工作流模式）、Phase 14 · 25（多智能体辩论）
**预计时间：** 约 60 分钟

## 学习目标

- 列出四种反复出现的编排模式及各自适用场景。
- 描述 2026 年 LangChain 的建议：基于工具调用的监督与监督者库。
- 解释 Anthropic 的"构建正确系统"规则以及它如何约束拓扑选择。
- 在标准库中针对共同的脚本化 LLM 实现全部四种模式。

## 问题背景

团队在真正需要之前就选择"多智能体"。四种模式在框架中反复出现；一旦你能命名它们，就可以选择正确的——或者完全跳过拓扑。

## 核心概念

### 监督者-工作者

- 中心路由 LLM 分发给专家智能体。
- 决定：循环回自身、切换给专家、终止。
- 专家之间不互相通信；所有路由都经过监督者。

框架：LangGraph `create_supervisor`、Anthropic 编排器-工作者、CrewAI 层次化流程。

**2026 年 LangChain 建议：** 通过直接工具调用而不是 `create_supervisor` 来实现监督。提供更精细的上下文工程控制——你精确决定每个专家看到什么。

### 蜂群/点对点

- 智能体通过共享工具界面直接切换。
- 没有中心路由器。
- 比监督者延迟更低（更少跳数）。
- 更难推理（没有单一控制点）。

框架：LangGraph 蜂群拓扑、OpenAI Agents SDK 切换（当所有智能体都可以切换给其他所有智能体时）。

### 层次化

- 监督者管理子监督者，子监督者管理工作者。
- 在 LangGraph 中实现为嵌套子图；在 CrewAI 中实现为嵌套 crew。
- 以运维复杂性为代价，扩展到大型智能体群体。

何时需要它：当单个监督者的上下文预算无法容纳所有专家的描述时。

### 辩论

- 并行提案者 + 迭代交叉批评（第 25 课）。
- 严格来说不算编排——更像验证——但在框架中作为拓扑选择出现。

### CrewAI Crew 与 Flow

CrewAI 规范化了两种部署模式：

- **Flow** 用于确定性事件驱动自动化（推荐的生产起点）。
- **Crew** 用于自主角色协作。

这与上述四种模式正交，但映射到拓扑：Flow 通常是监督者或层次化；Crew 通常是带 LLM 路由器的监督者。

### Anthropic 的指导原则

"LLM 领域的成功不在于构建最复杂的系统。而在于为你的需求构建正确的系统。"

决策顺序：

1. 单个智能体 + 工作流模式（第 12 课）——从这里开始。
2. 监督者-工作者——当你有 2-4 个专家时。
3. 蜂群——当延迟比推理清晰度更重要时。
4. 层次化——只有当监督者上下文预算不足时。
5. 辩论——当准确率比成本更重要时。

### 这个模式在哪里出错

- **拓扑优先思维。** 在识别出多智能体解决什么问题之前就说"我们需要多智能体"。
- **蜂群中的反弹切换。** A -> B -> A -> B。使用跳数计数器。
- **虚假层次。** 三层因为"企业要求"；实际上两个团队。压缩掉。

## 动手实践

`code/main.py` 在标准库中针对脚本化 LLM 实现了全部四种模式：

- `Supervisor`——中心路由器。
- `Swarm`——带直接切换的点对点。
- `Hierarchical`——监督者的监督者。
- `Debate`——并行提案者 + 批评。

每种模式处理相同的三意图任务（退款/bug/销售）。追踪形态不同。

运行：

```
python3 code/main.py
```

输出：每种模式的追踪 + 操作数。监督者最清晰；蜂群最短；层次化最深；辩论最昂贵。

## 使用建议

- **LangGraph** 用于监督者和层次化（嵌套子图）。
- **OpenAI Agents SDK** 用于作为工具的切换（监督者形态）。
- **CrewAI Flow** 用于生产确定性场景。
- **自定义** 用于辩论或需要精确控制时。

## 练习

1. 通过移除路由器将监督者-工作者转换为蜂群。什么出了问题？什么改善了？
2. 向蜂群添加跳数计数器：3 次切换后拒绝。它能捕获 A->B->A 反弹吗？
3. 为 12 个专家的领域构建两级层次化系统。没有嵌套的情况下，上下文预算在哪里失败？
4. 在生产形态的工作负载上剖析四种模式。哪种在哪个指标上胜出（延迟、成本、准确率、可调试性）？
5. 阅读 Anthropic 的"构建有效智能体"文章。将你的每个生产流程映射到四种模式之一。有没有不能清晰映射的？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 监督者-工作者 | "路由器 + 专家" | 中心 LLM 分发给专家；专家之间不通信 |
| 蜂群 | "点对点" | 通过共享工具直接切换；无中心路由器 |
| 层次化 | "监督者的监督者" | 用于大型群体的嵌套子图 |
| 辩论 | "提案者 + 批评" | 并行提案者，交叉批评（第 25 课） |
| 基于工具调用的监督 | "无库的监督者" | 将监督者实现为直接工具调用以控制上下文 |
| Crew | "自主团队" | CrewAI 的基于角色的协作模式 |
| Flow | "确定性工作流" | CrewAI 的事件驱动生产模式 |
