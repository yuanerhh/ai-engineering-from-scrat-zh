# 智能体可观测性：Langfuse、Phoenix、Opik

> 三个开源智能体可观测性平台主导 2026 年。Langfuse（MIT）——每月 600 万+ 次安装，追踪 + 提示词管理 + 评估 + 会话重放。Arize Phoenix（Elastic 2.0）——深度智能体专项评估、RAG 相关性、OpenInference 自动检测。Comet Opik（Apache 2.0）——自动化提示词优化、防护栏、LLM 判断器幻觉检测。

**类型：** 学习
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 23（OTel GenAI）
**预计时间：** 约 45 分钟

## 学习目标

- 列出三大开源智能体可观测性平台及其许可证。
- 区分各平台的优势所在：Langfuse（提示词管理 + 会话）、Phoenix（RAG + 自动检测）、Opik（优化 + 防护栏）。
- 解释为什么到 2026 年 89% 的组织报告已部署智能体可观测性。
- 实现一个带 LLM 判断器评估的标准库追踪到仪表板流水线。

## 问题背景

OTel GenAI（第 23 课）提供了 schema。你仍然需要能摄入 span、运行评估、存储提示词版本并呈现回归的平台。三个竞争者各自强调生命周期的不同部分。

## 核心概念

### Langfuse（MIT）

- 每月 600 万+ SDK 安装量，19k+ GitHub stars。
- 功能：追踪、带版本控制 + 游乐场的提示词管理、评估（LLM 作为判断器、用户反馈、自定义）、会话重放。
- 2025 年 6 月：原先的商业模块（LLM 作为判断器、注释队列、提示词实验、游乐场）在 MIT 下开源。
- 最强于：带紧密提示词管理循环的端到端可观测性。

### Arize Phoenix（Elastic License 2.0）

- 更深入的智能体专项评估：追踪聚类、异常检测、RAG 检索相关性。
- 原生 OpenInference 自动检测。
- 与托管 Arize AX 配对用于生产环境。
- 无提示词版本控制——定位为与更广泛平台配合使用的漂移/行为回归工具。
- 最强于：RAG 相关性、行为漂移、异常检测。

### Comet Opik（Apache 2.0）

- 通过 A/B 实验进行自动化提示词优化。
- 防护栏（PII 脱敏、主题约束）。
- LLM 判断器幻觉检测。
- Comet 自身测量的基准：Opik 在 23.44 秒内完成日志 + 评估，而 Langfuse 需要 327.15 秒（约 14 倍差距）——将供应商基准视为方向性指标。
- 最强于：优化循环、自动化实验、防护栏执行。

### 行业数据

根据 Maxim（2026 年实地分析）：89% 的组织已部署智能体可观测性；质量问题是主要的生产障碍（32% 的受访者提及）。

### 如何选择

| 需求 | 选择 |
|------|------|
| 带提示词管理的一体化方案 | Langfuse |
| 深度 RAG 评估 + 漂移检测 | Phoenix |
| 自动化优化 + 防护栏 | Opik |
| 开放许可证，无 ELv2 | Langfuse（MIT）或 Opik（Apache 2.0） |
| Datadog / New Relic 集成 | 任意——它们都导出 OTel |

### 这个模式在哪里出错

- **没有评估策略。** 没有评估的追踪只是昂贵的日志记录。
- **没有依据的自制 LLM 判断器。** CRITIC 模式（第 05 课）适用——判断器需要外部工具进行事实核查。
- **提示词版本未与追踪关联。** 当生产环境出现回归时，你无法定位到导致问题的提示词。

## 动手实践

`code/main.py` 实现了一个标准库追踪收集器 + LLM 判断器评估器：

- 摄入 GenAI 形状的 span。
- 按会话分组，标记失败运行（防护栏触发、低置信度评估）。
- 一个脚本化的 LLM 判断器，根据评分标准对智能体响应打分。
- 类仪表板摘要：失败率、主要失败原因、评估分数分布。

运行：

```
python3 code/main.py
```

输出：与 Langfuse/Phoenix/Opik 显示内容匹配的每会话评估分数和失败分类。

## 使用建议

- **Langfuse** 自托管或云端；通过 OTel 或其 SDK 接入。
- **Arize Phoenix** 自托管；自动检测 OpenInference。
- **Comet Opik** 自托管或云端；自动化优化循环。
- **Datadog LLM 可观测性** 适用于已运行 Datadog 的混合运维+ML 团队。

`outputs/skill-obs-platform-wiring.md` 选择一个平台，将追踪 + 评估 + 提示词版本接入现有智能体。

## 练习

1. 将一周的 OTel 追踪导出到 Langfuse 云端（免费套餐）。哪些会话失败了？为什么？
2. 为你的领域编写 LLM 判断器评分标准（事实准确性、语气、范围遵从性）。在 50 条追踪上测试。
3. 比较 Langfuse 提示词版本控制与 Phoenix 的追踪聚类。哪个能更快告诉你什么出了问题？
4. 阅读 Opik 的防护栏文档。为你的某个智能体运行接入 PII 脱敏防护栏。
5. 在你的语料库上基准测试三者。忽略供应商发布的数字；自己测量。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 追踪 | "Span 收集器" | 摄入 OTel / SDK span；按会话索引 |
| 提示词管理 | "提示词 CMS" | 与追踪关联的版本化提示词 |
| LLM 作为判断器 | "自动化评估" | 独立 LLM 根据评分标准对智能体输出打分 |
| 会话重放 | "追踪回放" | 逐步回溯历史运行以调试 |
| RAG 相关性 | "检索质量" | 检索到的上下文是否与查询匹配 |
| 追踪聚类 | "行为分组" | 聚类相似运行以进行漂移检测 |
| 防护栏执行 | "日志时策略检查" | 对记录内容进行 PII/毒性/范围检查 |
