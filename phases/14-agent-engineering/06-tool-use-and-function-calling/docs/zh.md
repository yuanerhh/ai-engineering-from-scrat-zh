# 工具使用与函数调用

> Toolformer（Schick 等人，2023）开创了自监督的工具注释方法。伯克利函数调用排行榜 V4（Patil 等人，2025）设定了 2026 年的基准：40% 智能体类、30% 多轮、10% 实时、10% 非实时、10% 幻觉检测。单轮已接近解决。记忆、动态决策和长视野工具链尚未解决。

**类型：** 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 01（智能体循环）、Phase 13 · 01（函数调用深度解析）
**预计时间：** 约 60 分钟

## 学习目标

- 解释 Toolformer 的自监督训练信号：只在执行能降低下一个 token 损失时保留工具注释。
- 列出 BFCL V4 的五个评估类别及其测量内容。
- 用标准库实现一个带 schema 验证、参数强制转换和执行沙箱的工具注册表。
- 诊断 2026 年三个未解决的问题：长视野工具链、动态决策和记忆。

## 问题背景

早期的工具使用问的是：模型能预测出正确的函数调用吗？现代工具使用问的是：模型能在 40 步中链式调用工具，带着记忆、在部分可观察性下、从工具失败中恢复、且不幻觉出不存在的工具吗？

Toolformer 建立了基准：模型可以通过自监督学习何时调用工具。BFCL V4 定义了 2026 年的评估目标。两者之间的差距正是生产智能体所处的空间。

## 核心概念

### Toolformer（Schick 等人，NeurIPS 2023）

思路：让模型用候选 API 调用对自己的预训练语料库进行注释。对每个候选项，执行它。只有在包含工具结果能降低下一个 token 的损失时才保留注释。在过滤后的语料库上进行微调。

涵盖的工具：计算器、问答系统、搜索引擎、翻译器、日历。自监督信号纯粹关于工具是否有助于预测文本——没有人工标注。

规模结果：工具使用能力随规模出现。较小的模型因工具注释而变差；较大的模型受益。这就是为什么 2026 年的前沿模型内置了强大的工具使用能力，而大多数 7B 模型需要显式的工具使用微调才能可靠运行。

### 伯克利函数调用排行榜 V4（Patil 等人，ICML 2025）

BFCL 是 2026 年事实上的评估标准。V4 构成：

- **智能体类（40%）**——完整的智能体轨迹：记忆、多轮、动态决策。
- **多轮（30%）**——带工具链的交互式对话。
- **实时（10%）**——用户提交的真实提示词（更难的分布）。
- **非实时（10%）**——合成测试用例。
- **幻觉检测（10%）**——检测何时不应该调用工具。

V3 引入了基于状态的评估：在工具序列之后，检查 API 的实际状态（如"文件创建了吗？"），而不是匹配工具调用的 AST。V4 添加了网络搜索、记忆和格式敏感性类别。

2026 年的关键发现：单轮函数调用几乎已解决。失败集中在记忆（跨轮次携带上下文）、动态决策（根据先前结果选择工具）、长视野链（20 步以上的漂移）和幻觉检测（在没有合适工具时拒绝调用）方面。

### 工具 schema

每个提供商都有其 schema。细节不同但形状相同：

```
name: 字符串
description: 字符串（它做什么，何时使用）
input_schema: JSON Schema（属性、必需字段、类型、枚举）
```

Anthropic 直接使用 `input_schema`。OpenAI 使用 `function.parameters`。两者都接受 JSON Schema。描述是关键所在——模型读取它来选择正确的工具。糟糕的工具描述是"选错工具"失败的头号根本原因。

### 参数验证

永远不要信任工具调用。验证：

1. **类型强制转换。** 模型可能在 schema 要求 int 的地方返回字符串 "5"。如果无歧义则强制转换；否则拒绝。
2. **枚举验证。** 如果 schema 说 `status in {"open", "closed"}` 而模型输出 `"in_progress"`，拒绝并给出描述性错误。
3. **必需字段。** 缺少必需字段 -> 立即将错误观察返回给模型，而不是崩溃。
4. **格式验证。** 日期、邮箱、URL——用具体解析器验证，不用正则表达式。

每次验证失败都应该返回结构化的观察，使模型能够用正确的形态重试。

### 并行工具调用

现代提供商支持在一个助手轮次中并行工具调用。循环：

1. 模型发出 3 个带不同 `tool_use_id` 的工具调用。
2. 运行时（在独立时并行）执行它们。
3. 每个结果作为与 `tool_use_id` 关联的 `tool_result` 块返回。

工程规则：将关联 ID 视为关键所在。交换它们会导致错误的工具结果路由到错误的工具。

### 沙箱

工具执行是沙箱边界。详见第 09 课。简短版本：每个工具应该指定读/写范围、网络访问、超时和内存上限。通用的 `run_shell(cmd)` 是红旗；具体的 `git_status()` 更安全。

## 动手实践

`code/main.py` 实现了一个生产形态的工具注册表：

- JSON Schema 子集验证器（仅标准库）。
- 带描述、输入 schema、超时和执行器的工具注册。
- 参数强制转换和枚举验证。
- 带关联 ID 的并行工具分发。
- 以结构化字符串形式输出的错误观察。

运行：

```
python3 code/main.py
```

轨迹显示一个迷你智能体在一轮中调用三个工具，其中一个故意格式错误的调用被拒绝，并附有模型可以据此行动的描述性错误。

## 使用建议

每个提供商都有自己的工具 schema——Anthropic、OpenAI、Gemini、Bedrock。如果需要多提供商支持，使用转换层（OpenAI Agents SDK、Vercel AI SDK、LangChain 工具适配器）。BFCL 是参考基准——如果工具使用是产品的核心，在发布前对你的智能体运行它。

## 产出技能

`outputs/skill-tool-registry.md` 为给定任务领域生成工具目录、schema 和注册表。包括描述质量检查（每个工具的描述是否告诉模型何时使用它？）。

## 练习

1. 添加一个"空操作"工具，让模型明确拒绝使用任何其他工具。在类 BFCL 幻觉检测测试上进行测量。
2. 实现 int-as-string 和 float-as-string 的参数强制转换。强制转换在什么地方开始隐藏真正的 bug？
3. 为每个工具添加超时和熔断器（3 次连续失败后拒绝该工具 60 秒）。这如何改变模型的恢复方式？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 函数调用 | "工具使用" | 带验证 schema 的结构化输出工具调用 |
| Toolformer | "自监督工具注释" | Schick 2023——保留使结果降低下一个 token 损失的工具调用 |
| BFCL | "伯克利函数调用排行榜" | 2026 基准：40% 智能体类、30% 多轮、10% 实时、10% 非实时、10% 幻觉 |
| 工具 schema | "模型的函数签名" | 名称、描述、参数的 JSON Schema |
| tool_use_id | "关联 ID" | 将工具调用与其结果绑定；并行分发必不可少 |
| 幻觉检测 | "知道何时不调用" | V4 类别：在没有合适工具时拒绝调用 |
| 参数强制转换 | "字符串转整型修复" | 对可预测的 schema 不匹配进行窄修复；模糊时拒绝 |
| 沙箱 | "工具执行边界" | 每工具的读/写范围、网络、超时、内存上限 |
