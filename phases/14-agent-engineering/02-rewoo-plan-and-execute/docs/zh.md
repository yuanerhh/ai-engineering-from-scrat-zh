# ReWOO 与计划-执行：解耦规划

> ReAct 将思考和行动交织在一个流中。ReWOO 将它们分开：先做一个完整的计划，然后执行。token 减少 5 倍，HotpotQA 精度提升 +4%，并且可以将规划器精炼为 7B 模型。Plan-and-Execute 对其进行了推广；Plan-and-Act 将其扩展到网页导航。

**类型：** 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 01（智能体循环）
**预计时间：** 约 60 分钟

## 学习目标

- 解释为什么 ReWOO 的规划器（Planner）/ 工作器（Worker）/ 求解器（Solver）分离相比 ReAct 的交织循环节省了 token 并提高了鲁棒性。
- 实现一个计划 DAG（有向无环图）、一个依赖序执行器和一个组合工作器输出的求解器——全部使用标准库。
- 使用 2026 年"五种工作流模式"框架（Anthropic）来决定任务应以先计划后执行还是交织 ReAct 的方式运行。
- 识别何时需要 Plan-and-Act 的合成计划数据，用于长视野的网页或移动任务。

## 问题背景

ReAct 的交织思考-行动-观察循环简单且灵活，但每次工具调用都必须携带完整的先前上下文——包括之前的每一个思考。token 使用量随深度呈二次增长。更糟糕的是：当工具在循环中途失败时，模型必须从错误观察中重新推导整个计划。

ReWOO（Xu 等人，arXiv:2305.18323，2023 年 5 月）注意到了这个问题，并做了一个决策：预先规划整件事，并行获取证据，最后组合答案。一次 LLM 调用来规划，N 次工具调用获取证据（可以并行），一次 LLM 调用来求解。代价是灵活性降低（计划是静态的），换来大幅提升的 token 效率和更清晰的失败模式。

## 核心概念

### 三个角色

```
Planner（规划器）:  用户问题 -> [计划 DAG]
Workers（工作器）:  [计划 DAG] -> [证据]     （工具调用，可能并行）
Solver（求解器）:   用户问题, 计划 DAG, 证据 -> 最终答案
```

规划器生成一个 DAG。每个节点命名一个工具、其参数，以及它依赖哪些更早的节点（如 `#E1`、`#E2` 这样的引用）。工作器按拓扑顺序执行节点。求解器将所有内容拼合在一起。

### 为什么减少 5 倍 token

ReAct 的提示词长度随步骤数线性增长。到第 10 步时，提示词包含思考 1、行动 1、观察 1、思考 2、行动 2、观察 2……每个中间步骤还冗余地包含原始提示词。

ReWOO 的开销：一次规划器提示词（较大），N 个小型工作器提示词（每个只有工具调用，没有链式上下文），一次求解器提示词。论文在 HotpotQA 上测量到约 5 倍更少的 token，同时绝对精度提升 +4 个点。

### 为什么更鲁棒

如果 ReAct 中的工作器 3 失败，循环必须在流的中途从错误中推理出路。在 ReWOO 中，工作器 3 返回一个错误字符串；求解器在上下文中结合原始计划看到它，可以优雅地降级处理。失败定位是按节点的，而不是按步骤的。

### 规划器精炼

论文的第二个结果：由于规划器不看观察结果，你可以在 175B 教师模型的规划器输出上微调一个 7B 模型。小模型负责规划；推理时不需要大模型。这现在已成为标准——2026 年很多生产智能体使用小型规划器和大型执行器，或反之。

### Plan-and-Execute（LangChain，2023 年）

LangChain 团队 2023 年 8 月的文章将 ReWOO 推广为一种模式名称：Plan-and-Execute（计划与执行）。预先规划器输出步骤列表，执行器运行每个步骤，可选的再规划器可以在观察结果后修订计划。这比 ReWOO 更接近 ReAct（再规划器将观察结果带回规划），但保留了 token 节省的优势。

### Plan-and-Act（Erdogan 等人，arXiv:2503.09572，ICML 2025）

Plan-and-Act 将该模式扩展到长视野的网页和移动智能体。关键贡献是合成计划数据：一个带标注的轨迹生成器生成计划是明确的训练数据。用于微调规划器模型，使其在 WebArena 类任务上能跑过 30-50 步，而单一 ReAct 轨迹在这里会失去连贯性。

### 如何选择

| 模式 | 适用场景 |
|------|---------|
| ReAct | 短任务、未知环境、需要响应式异常处理 |
| ReWOO | 已知工具的结构化任务、对 token 敏感、证据可并行获取 |
| Plan-and-Execute | 类似 ReWOO 但需要在部分执行后重新规划 |
| Plan-and-Act | 长视野（>30 步）、网页/移动/计算机使用 |
| Tree of Thoughts | 搜索值得付出代价时（第 04 课） |

Anthropic 2024 年 12 月的指导：从最简单的开始。如果任务是一次工具调用加上一个摘要，不要构建 ReWOO。如果任务是一个 40 步的研究任务，不要单独使用 ReAct。

## 动手实践

`code/main.py` 实现了一个玩具 ReWOO：

- `Planner`（规划器）——一个从提示词输出计划 DAG 的脚本化策略。
- `Worker`（工作器）——通过注册表分发每个节点的工具调用。
- `Solver`（求解器）——读取证据并生成最终答案的脚本化组合。
- 依赖解析——像 `#E1` 这样的引用在分发时被替换为更早的工作器输出。

演示通过两步计划回答"法国首都的人口，四舍五入到百万是多少？"：（1）查找首都，（2）查找人口，然后求解。

运行：

```
python3 code/main.py
```

轨迹首先显示完整计划，然后是工作器结果，然后是求解器组合。将 token 计数（我们打印粗略的字符数）与 ReAct 风格的交织运行进行比较——ReWOO 在这类结构化任务上更胜一筹。

## 使用建议

LangGraph 将 Plan-and-Execute 作为配方提供（`create_react_agent` 用于 ReAct，自定义图用于计划执行）。CrewAI 的 Flows 直接编码了该模式：你预先定义任务，Flow DAG 执行它们。Plan-and-Act 的合成数据方法仍主要是研究性的；运行时模式（显式计划 DAG）通过 LangGraph 和 CrewAI Flows 在生产中应用。

## 产出技能

`outputs/skill-rewoo-planner.md` 根据用户请求和工具目录生成 ReWOO 计划 DAG。在移交给执行器之前验证计划（无环、每个引用都已解析、每个工具都存在）。

## 练习

1. 并行化独立计划节点的工作器执行。在一个有 2 个并行组的 6 节点 DAG 上能得到什么收益？
2. 添加一个再规划节点，在任何工作器返回错误时触发。将 ReWOO 变成 Plan-and-Execute 最小需要改什么？
3. 将 `Planner` 替换为小型模型（7B 级），保持 `Solver` 使用前沿模型。比较端到端质量——分离在哪里失败？
4. 阅读 ReWOO 论文第 4 节关于规划器精炼的内容。从概念上复现 175B -> 7B 结果：你需要什么训练数据？如何评分计划质量？
5. 将玩具改造为 Plan-and-Act 的轨迹形态：计划是序列而非 DAG。有哪些权衡发生了变化？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| ReWOO | "无观察的推理" | 先计划，然后并行获取证据，最后求解——规划提示词中没有观察结果 |
| Plan-and-Execute | "LangChain 的计划执行模式" | 带可选再规划节点的 ReWOO |
| Plan-and-Act | "扩展的计划执行" | 带合成计划训练数据的显式规划器/执行器分离，用于长视野任务 |
| 证据引用 | "#E1, #E2, ..." | 计划节点占位符，在分发时替换为先前工作器输出 |
| 规划器精炼 | "小规划器，大执行器" | 在大型教师模型的规划器轨迹上微调小型模型 |
| Token 效率 | "更少的来回" | 论文中相比 ReAct 在 HotpotQA 上减少 5 倍 token |
| DAG 执行器 | "拓扑调度器" | 按依赖顺序运行计划节点；每层并行 |
