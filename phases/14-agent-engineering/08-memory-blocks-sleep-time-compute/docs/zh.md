# 记忆块与睡眠时间计算（Letta）

> MemGPT 在 2024 年变成了 Letta。2026 年的演进新增了两个思路：模型可以直接编辑的离散功能性记忆块，以及一个在主智能体空闲时异步整合记忆的睡眠时间智能体。这就是如何将记忆扩展到单个对话之外的方法。

**类型：** 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 07（MemGPT）
**预计时间：** 约 75 分钟

## 学习目标

- 说出 Letta 使用的三个记忆层（核心、召回、归档）及各自的作用。
- 解释记忆块模式：Human 块、Persona 块和用户自定义块作为一等类型化对象。
- 描述睡眠时间计算是什么、为什么它位于关键路径之外，以及为什么它可以运行比主智能体更强的模型。
- 实现一个脚本化的双智能体循环：主智能体提供响应，睡眠时间智能体在轮次之间整合块。

## 问题背景

MemGPT（第 07 课）解决了虚拟内存控制流。随后出现了三个生产问题：

1. **延迟。** 每次记忆操作都在关键路径上。如果智能体在用户等待时必须剪枝、摘要或协调，尾延迟会激增。
2. **记忆腐化。** 写入积累。被推翻的事实留存下来。检索被陈旧内容淹没。
3. **结构丢失。** 扁平的归档存储无法表达"Human 块始终在提示词中；Persona 块始终在提示词中；Task 块按会话交换"。

Letta（letta.com）是 2026 年的重写版本。记忆块使结构变得明确；睡眠时间计算将整合移出关键路径。

## 核心概念

### 三个层次

| 层次 | 范围 | 存储位置 | 写入者 |
|------|-----|---------|-------|
| 核心（Core） | 始终可见 | 在主提示词内 | 智能体工具调用 + 睡眠时间重写 |
| 召回（Recall） | 对话历史 | 可检索 | 自动轮次记录 |
| 归档（Archival） | 任意事实 | 向量 + KV + 图 | 智能体工具调用 + 睡眠时间摄入 |

核心是 MemGPT 的核心。召回是带驱逐尾巴的对话缓冲区。归档是外部存储。这个拆分清理了 MemGPT 二级的混乱。

### 记忆块

块是核心层的类型化、持久化、可编辑节。原始 MemGPT 论文定义了两个：

- **Human 块**——关于用户的事实（姓名、角色、偏好、目标）。
- **Persona 块**——智能体的自我概念（身份、语气、约束）。

Letta 推广到任意用户自定义块：当前目标的 `Task` 块、代码库事实的 `Project` 块、硬约束的 `Safety` 块。每个块有 `id`、`label`、`value`、`limit`（字符上限）、`description`（让模型知道何时编辑它）。

块可通过工具接口编辑：

- `block_append(label, text)`
- `block_replace(label, old, new)`
- `block_read(label)`
- `block_summarize(label)`——对接近上限的块进行压缩。

### 睡眠时间计算

2025 年的 Letta 新增功能：在后台运行第二个智能体，位于关键路径之外。睡眠时间智能体处理对话记录和代码库上下文，将 `learned_context`（学到的上下文）写入共享块，整合或使归档记录失效。

产生的特性：

- **无延迟成本。** 主响应不等待记忆操作。
- **允许更强的模型。** 睡眠时间智能体可以是更昂贵、更慢的模型，因为它没有延迟约束。
- **自然整合窗口。** 在用户不等待时去重、摘要、使被推翻的事实失效。

这个形态与人类工作方式相符：完成任务，睡一觉，长期记忆在夜间沉淀。

### Letta V1 与原生推理

Letta V1（`letta_v1_agent`，2026）废弃了 `send_message`/心跳和内联 `Thought:` token，转而使用原生推理。Responses API（OpenAI）和带扩展思考的 Messages API（Anthropic）在独立通道上输出推理，跨轮次传递（在生产中跨提供商加密）。控制循环仍然是 ReAct。思考轨迹是结构化的，而非提示词形态的。

### 这个模式在哪里出错

- **块膨胀。** 无限的 `block_append` 很快触达上限。在超出上限的写入之前接上块摘要器。
- **静默漂移。** 睡眠时间智能体重写了一个块，主智能体从未注意到。对块进行版本控制，在轨迹中显示差异。
- **投毒整合。** 睡眠时间智能体将攻击者可达的内容处理进核心。第 27 课也适用于睡眠时间表面。

## 动手实践

`code/main.py` 实现了：

- `Block`——id、label、value、limit、description。
- `BlockStore`——CRUD + `near_limit(label)` 辅助函数。
- 两个脚本化智能体——`PrimaryAgent` 提供轮次，`SleepTimeAgent` 在轮次之间整合。
- 一个展示三轮对话的轨迹，带有块写入，以及一个对块进行摘要并使陈旧事实失效的睡眠时间传递。

运行：

```
python3 code/main.py
```

记录显示了分离：主轮次快速产生原始写入；睡眠传递压缩并清理。

## 使用建议

- **Letta**（letta.com）用于参考实现。自托管或托管云。
- **Claude Agent SDK 技能**作为块形态的知识——技能是一个智能体按需加载的命名、版本化、可检索的指令块。
- **自定义构建**用于需要控制存储后端的团队。使用 Letta API 契约，这样以后可以迁移。

## 产出技能

`outputs/skill-memory-blocks.md` 为任何运行时生成 Letta 形态的块系统，带睡眠时间钩子，包括安全规则和引用连接。

## 练习

1. 添加一个 `block_summarize` 工具，当 `near_limit` 返回 true 时用模型生成的摘要替换块值。哪个触发阈值既能最小化摘要调用又能防止块溢出？
2. 在归档上实现睡眠时间去重：文本 token 重叠 >90% 的两条记录合并为一条。只在睡眠传递中做，永远不在关键路径上。
3. 对块进行版本控制。在每次写入时记录旧值和差异。暴露 `block_history(label)` 使操作人员可以调试"智能体为什么忘记了 X"。
4. 将睡眠时间智能体视为不受信任的写入者。当它们触及 Persona 或 Safety 块时，在提交之前需要第二个智能体审查。
5. 将示例移植到使用 Letta API（`letta_v1_agent`）。块 schema 有什么变化？原生推理如何改变轨迹形态？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 记忆块（Memory block） | "可编辑提示词节" | 类型化、持久化、LLM 可编辑的核心记忆段 |
| Human 块 | "用户记忆" | 关于用户的事实，固定在核心中 |
| Persona 块 | "智能体身份" | 自我概念、语气、约束，固定在核心中 |
| 睡眠时间计算 | "异步记忆工作" | 第二个智能体在关键路径之外进行整合 |
| 核心/召回/归档 | "层次" | 三层记忆拆分：始终可见 / 对话 / 外部 |
| 块上限（Block limit） | "上限" | 每块字符限制；强制摘要 |
| 原生推理 | "思考通道" | 提供商级别的推理输出，而非提示词级别的 `Thought:` |
| 学到的上下文 | "睡眠输出" | 睡眠时间智能体写入共享块的事实 |
