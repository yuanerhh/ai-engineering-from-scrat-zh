# 混合记忆：向量 + 图 + KV（Mem0）

> Mem0（Chhikara 等人，2025）将记忆视为三个并行存储——向量用于语义相似度，KV 用于快速事实查询，图用于实体-关系推理。检索时一个评分层融合三者。这是 2026 年外部记忆的生产标准。

**类型：** 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 07（MemGPT）、Phase 14 · 08（Letta 块）
**预计时间：** 约 75 分钟

## 学习目标

- 解释为什么单一存储（仅向量、仅图或仅 KV）不足以支持智能体记忆。
- 列出 Mem0 的三个并行存储及各自优化的方向。
- 描述 Mem0 的融合评分——相关性、重要性、时效性——以及为什么是加权和而非层级结构。
- 用标准库实现一个玩具三存储记忆，其中 `add()` 写入全部三个，`search()` 融合结果。

## 问题背景

单一存储对三类查询中的一类来说总是错误的：

- **语义相似度**——"上周我们讨论了什么关于智能体漂移的内容？"向量胜出；KV 和图遗漏。
- **事实查询**——"用户的电话号码是什么？"KV 胜出；向量是浪费，图是过度设计。
- **关系推理**——"哪些客户共享相同的计费实体？"图胜出；向量和 KV 无法回答。

生产智能体在一次会话中会发出全部三类查询。单一存储记忆对其中两类总是错误的。Mem0 的贡献是将三者连接在一个单一的 `add`/`search` 接口后面，并用融合评分函数统一它们。

## 核心概念

### 三个并行存储

Mem0（arXiv:2504.19413，2025 年 4 月）在 `add(text, user_id, metadata)` 时：

1. 从文本中提取候选事实（LLM 驱动的步骤）。
2. 将每个事实写入向量存储（嵌入）用于语义搜索。
3. 将每个事实写入 KV 存储，以 (user_id, fact_type, entity) 为键进行 O(1) 查询。
4. 将每个事实写入图存储（Mem0g）作为类型化边用于关系查询。

在 `search(query, user_id)` 时：

1. 向量存储按嵌入余弦返回 top-k。
2. KV 存储返回以查询派生的 (user_id, type, entity) 为键的直接命中。
3. 图存储返回从查询实体可达的子图。
4. 评分层融合三者。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **相关性（Relevance）**——向量余弦、KV 精确匹配、图路径权重。
- **重要性（Importance）**——在写入时标记或学习（某些事实更重要：姓名、ID、策略）。
- **时效性（Recency）**——自上次写入或读取以来基于时间的指数衰减。

权重按产品调整。聊天智能体的 `w_recency` 更高；合规智能体的 `w_importance` 更高；检索智能体的 `w_relevance` 更高。

### Mem0g 与时态推理

Mem0g 添加了冲突检测器。当新事实与现有边矛盾时，现有边被标记为无效但不删除。时态查询（"用户三月份的城市是什么？"）遍历在该时间点有效的子图。

这是 Letta 的失效模式泛化的合规级行为。

### 基准数字

Mem0 论文报告（2025 年）：

- **LoCoMo**（长格式对话记忆）：91.6
- **LongMemEval**（长视野情节记忆）：93.4
- **BEAM 1M**（100 万 token 记忆基准）：64.1

对比基准（全上下文 128k LLM、扁平向量存储、扁平 KV）全部差 10+ 分。基准数字本身不足以证明选择的合理性——操作形态才是——但数字表明融合设计不是可以忽略的误差。

### 范围分类

Mem0 按范围划分记忆：

- **用户记忆（User memory）**——跨会话持久化，以 `user_id` 为键。
- **会话记忆（Session memory）**——在一个线程内持久化。
- **智能体记忆（Agent memory）**——每个智能体实例的状态。

每次写入选择一个范围。检索可以带每范围权重地跨范围查询。不加思考地混合范围会导致"助手把 Bob 的项目信息告诉了 Alice"这类事故。

### 这个模式在哪里出错

- **嵌入漂移。** 在前一百次查询中看起来正确的向量结果随着语料库增长而退化。定期对使用量前 N 的记录重新嵌入。
- **KV schema 蔓延。** `(user_id, type, entity)` 看起来简单，直到每个团队添加自己的 `type`。每季度审查类型集合。
- **图爆炸。** 一个嘈杂的提取器每条消息添加 50 条边。对每次 `add` 调用的图写入设上限；丢弃低置信度的边。

## 动手实践

`code/main.py` 用标准库实现了三存储模式：

- `VectorStore`——以朴素的 token 重叠相似度作为嵌入替代。
- `KVStore`——以 `(user_id, fact_type, entity)` 为键的字典。
- `GraphStore`——类型化边（主体、关系、客体、有效性）。
- `Mem0`——带 `add()`、`search()`、融合评分和范围感知检索的顶层门面。
- 多用户、多会话对话的完整轨迹。

运行：

```
python3 code/main.py
```

输出显示三条独立的召回路径以及融合后的 top-k。翻转 `main()` 顶部的评分权重，观察排名变化。

## 使用建议

- **Mem0（Apache 2.0）**——生产就绪。用 Postgres + Qdrant + Neo4j 自托管，或使用托管云。
- **Letta**——核心/召回/归档三层；自带向量和图后端。
- **Zep**——带时态知识图谱和事实提取的商业替代方案。
- **自定义构建**——当你需要对提取器（合规）或融合权重（语音智能体，时效性占主导）进行精确控制时。

## 练习

1. 将玩具向量相似度替换为真实的嵌入模型（sentence-transformers、Ollama、OpenAI 嵌入）。在合成的长对话上测量 recall@10。排名在 1000 次写入后是否漂移？
2. 添加时态查询：`search(query, as_of=timestamp)`。只返回在该时间或之前有效的记录。哪个存储需要最多改动？
3. 实现冲突检测器：如果传入的事实与图边矛盾，使旧边失效并记录两者。在"用户住在柏林" -> "用户住在里斯本"上测试。
4. 将融合评分器扩展以包含 `user_feedback` 维度（对检索记录点赞）。如何防止操纵（智能体只返回它已经喜欢的记录）？
5. 阅读 Mem0 文档（`docs.mem0.ai`）。将玩具移植到 `mem0` 客户端调用。在同样 20 个测试查询上比较检索质量。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 混合记忆 | "向量加图加 KV" | 三个并行写入的存储，检索时融合 |
| 事实提取 | "记忆摄入" | 将文本分解为（实体、关系、事实）元组的 LLM 步骤 |
| 融合评分 | "相关性排名" | 相关性、重要性、时效性的加权和 |
| 范围（Scope） | "记忆命名空间" | 用户/会话/智能体——决定谁看到什么 |
| Mem0g | "记忆图" | 带时态有效性的类型化边，用于关系查询 |
| 时态失效 | "软删除" | 标记矛盾的边为无效；永不删除 |
| 嵌入漂移 | "检索腐化" | 随语料库增长，向量质量退化；定期重新嵌入 |
