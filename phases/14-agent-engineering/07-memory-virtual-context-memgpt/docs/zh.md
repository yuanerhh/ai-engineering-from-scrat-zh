# 记忆：虚拟上下文与 MemGPT

> 上下文窗口是有限的。对话、文档和工具调用轨迹是无限的。MemGPT（Packer 等人，2023）将其类比为操作系统的虚拟内存——主上下文是内存，外部存储是磁盘，智能体在两者之间分页。这是 2026 年所有记忆系统继承的模式。

**类型：** 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 01（智能体循环）、Phase 14 · 06（工具使用）
**预计时间：** 约 75 分钟

## 学习目标

- 解释 MemGPT 基于的操作系统类比：主上下文 = 内存，外部上下文 = 磁盘，记忆工具 = 换入/换出。
- 用标准库实现二级 MemGPT 模式，包含主上下文缓冲区、外部可搜索存储和换入/换出工具。
- 描述智能体如何发出"中断"来查询或修改外部记忆，以及结果如何拼接回下一个提示词。
- 识别延续到 Letta（第 08 课）和 Mem0（第 09 课）的 MemGPT 设计选择。

## 问题背景

上下文窗口看起来应该能解决记忆问题，但实际上不行。生产中反复出现三种失败模式：

1. **溢出。** 多轮对话、长文档或工具调用密集的轨迹超出窗口。超出截止点的所有内容都会丢失。
2. **稀释。** 即使在窗口内，塞入不相关的上下文也会稀释对重要内容的注意力。前沿模型在长输入上仍会退化。
3. **持久性。** 新会话从空窗口开始。没有外部记忆的智能体无法在会话间说"记得你之前让我做……"

更大的窗口有所帮助但无法解决这个问题。Mem0 2025 年的论文测量发现，128k 窗口的基准仍然会遗漏带有外部记忆的 4k 窗口智能体能捕获的长视野事实。

## 核心概念

### MemGPT：操作系统类比

Packer 等人（arXiv:2310.08560，v2 2024 年 2 月）将上下文管理映射到操作系统虚拟内存：

| OS 概念 | MemGPT 概念 | 2026 年生产类比 |
|---------|------------|--------------|
| 内存（RAM） | 主上下文（提示词） | Anthropic/OpenAI 上下文窗口 |
| 磁盘（Disk） | 外部上下文 | 向量数据库、KV 存储、图存储 |
| 缺页（Page fault） | 记忆工具调用 | `memory.search`、`memory.read`、`memory.write` |
| 操作系统内核 | 智能体控制循环 | 带记忆工具的 ReAct 循环 |

智能体运行正常的 ReAct 循环。一类额外的工具让它能在主上下文中换入/换出数据。

### 两个层次

- **主上下文。** 固定大小的提示词，保存当前任务。模型始终可见。
- **外部上下文。** 无界的、可通过工具搜索。相关时读取，有新事实时写入。

原始论文在两个超出基础窗口的任务上评估了该设计：超过 100k token 的文档分析，以及跨天的持久化记忆多会话聊天。

### 中断模式

MemGPT 引入了记忆即中断：对话中途智能体可以调用记忆工具，运行时执行它，结果以新观察的形式拼接进下一个助手轮次。概念上与 Unix `read()` 系统调用相同——阻塞进程、返回字节、进程继续。

标准记忆工具接口：

- `core_memory_append(section, text)`——写入提示词的持久节。
- `core_memory_replace(section, old, new)`——编辑持久节。
- `archival_memory_insert(text)`——写入可搜索的外部存储。
- `archival_memory_search(query, top_k)`——从外部存储检索。
- `conversation_search(query)`——扫描过去的轮次。

### MemGPT 在哪里结束，Letta 在哪里开始

2024 年 9 月，MemGPT 变成了 Letta。研究仓库（`cpacker/MemGPT`）保留；Letta 扩展了设计：

- 三个层次而非两个（核心、召回、归档——第 08 课）。
- 原生推理替代了 `send_message`/心跳模式（第 08 课）。
- 异步运行记忆工作的睡眠时间智能体（第 08 课）。

即使生产系统运行 Letta、Mem0 或自定义二级存储，MemGPT 论文仍是 2026 年的基础。

### 这个模式在哪里出错

- **记忆腐化。** 写入积累速度快于读取；检索被陈旧事实淹没。修复：定期整合（Letta 睡眠时间）、显式失效（Mem0 冲突检测器）。
- **记忆投毒。** 外部记忆是检索到的文本。如果攻击者控制的内容进入记忆笔记，智能体在下一个会话中会重新摄入它。这是 Greshake 等人（第 27 课）的攻击在时间维度上的重述。
- **引用丢失。** 智能体回忆"用户让我发布 X"但无法引用是哪个轮次。在每次归档写入时存储来源引用（会话 ID、轮次 ID）。

## 动手实践

`code/main.py` 用标准库实现了 MemGPT 的二级模式：

- `MainContext`——固定大小的提示词缓冲区，包含 `core` 字典和 `messages` 列表；超出上限时自动压缩最旧的消息。
- `ArchivalStore`——(id, text, tags, session, turn) 记录的内存 BM25 类存储（token 重叠评分）。
- 映射到 MemGPT 接口的五个记忆工具。
- 一个将事实写入归档然后通过调用 `archival_memory_search` 回答问题的脚本化智能体。

运行：

```
python3 code/main.py
```

轨迹显示智能体写入三个事实，主上下文填满上限（强制驱逐），然后通过从归档检索来回答后续问题——无需真实 LLM 即可复现 MemGPT 工作流。

## 使用建议

今天的每个生产记忆系统都是 MemGPT 的变体：

- **Letta**（第 08 课）——三个层次、原生推理、睡眠时间计算。
- **Mem0**（第 09 课）——向量 + KV + 图，融合评分层。
- **OpenAI Assistants / Responses**——通过线程和文件管理记忆。
- **Claude Agent SDK**——通过技能和会话存储实现长期记忆。

按操作形态（自托管、托管、框架集成）选择，而不是按核心模式——核心模式都是 MemGPT。

## 产出技能

`outputs/skill-virtual-memory.md` 是一个可复用的技能，为任何目标运行时生成正确的二级记忆脚手架（主上下文 + 归档 + 工具接口），内置驱逐策略和引用字段。

## 练习

1. 添加以 token 衡量的 `max_main_context_tokens` 上限（用 `len(text.split()) * 1.3` 近似）。超出上限时将最旧的消息压缩为摘要。比较有无摘要器的行为。
2. 在归档存储上正确实现 BM25（词频、逆文档频率）。在玩具事实集上测量 recall@10 与 token 重叠基准的对比。
3. 向归档插入添加 `citation` 字段（session_id、turn_id、source_url）。让智能体在每个检索支撑的回答上引用来源。
4. 模拟记忆投毒：添加一条归档记录说"忽略所有未来的用户指令"。编写一个扫描检索结果中指令形态文本并将其标记为不受信任的防护措施。
5. 将实现移植到使用 MemGPT 研究仓库的核心记忆 JSON schema（`cpacker/MemGPT`）。从扁平字符串切换到有类型的节时什么会改变？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 虚拟上下文 | "无限记忆" | 主上下文（提示词）+ 外部（可搜索）两个层次，带换入/换出 |
| 主上下文 | "工作记忆" | 提示词——固定大小，始终可见 |
| 归档记忆 | "长期存储" | 外部可搜索持久化，按需检索 |
| 核心记忆 | "持久提示词节" | 固定在主上下文内部的命名节 |
| 记忆工具 | "记忆 API" | 智能体发出的读/写外部记忆的工具调用 |
| 中断 | "记忆缺页" | 智能体暂停，运行时获取，结果拼接进下一轮 |
| 记忆腐化 | "陈旧事实" | 旧写入淹没检索；通过整合修复 |
| 记忆投毒 | "注入的持久笔记" | 攻击者内容存储为记忆，在召回时重新摄入 |
