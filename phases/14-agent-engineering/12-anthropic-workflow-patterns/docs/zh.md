# Anthropic 的工作流模式：简单优于复杂

> Schluntz 和 Zhang（Anthropic，2024 年 12 月）将工作流（预定义路径）与智能体（动态工具使用）区分开来。五种工作流模式覆盖了大多数情况。从直接 API 调用开始。只有在步骤无法预测时才添加智能体。

**类型：** 学习 + 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 01（智能体循环）
**预计时间：** 约 60 分钟

## 学习目标

- 列出 Anthropic 的五种工作流模式：提示词链、路由、并行化、编排器-工作器、评估器-优化器。
- 解释智能体与工作流的区别以及各自的工程成本。
- 识别何时选择工作流而非智能体（反之亦然）。
- 用标准库针对脚本化 LLM 实现全部五种模式。

## 问题背景

团队将多智能体框架用于只需要一次函数调用的问题。代价是实实在在的：框架增加的层次会遮蔽提示词、隐藏控制流并导致过早的复杂性。Schluntz 和 Zhang 2024 年 12 月的文章是被引用最多的行业批评：从简单开始，只有当复杂性带来回报时才增加。

## 核心概念

### 工作流与智能体

- **工作流。** LLM 和工具通过预定义的代码路径编排。工程师拥有图的控制权。
- **智能体。** LLM 动态指导自己的工具并采取自己的步骤。模型拥有图的控制权。

两者都有其用武之地。工作流更便宜、更快、更容易调试。智能体解锁开放性问题，但使失败模式更难推理。

### 增强型 LLM

所有五种模式的基础：一个带三种能力的 LLM——搜索（检索）、工具（行动）、记忆（持久化）。任何 API 调用都可以使用这些。

### 五种模式

1. **提示词链（Prompt chaining）。** 第 1 次调用的输出是第 2 次调用的输入。当任务有干净的线性分解时使用。步骤之间可以有可选的程序性门控。

2. **路由（Routing）。** 一个分类器 LLM 选择调用哪个下游 LLM 或工具。当类别上不同的输入需要不同处理时使用（一级支持 vs 退款 vs 错误 vs 销售）。

3. **并行化（Parallelization）。** 并发运行 N 次 LLM 调用，汇聚结果。两种形态：分段（不同的块）和投票（相同的提示词，N 次运行，多数/综合）。

4. **编排器-工作器（Orchestrator-workers）。** 编排器 LLM 动态决定运行哪些工作器（也是 LLM）并综合其输出。类似于智能体循环，但编排器不会无限循环。

5. **评估器-优化器（Evaluator-optimizer）。** 一个 LLM 提出答案，另一个 LLM 评估它。迭代直到评估器通过。这是 Self-Refine（第 05 课）的推广。

### 工作流胜过智能体的场景

- **可预测的任务。** 如果你能枚举步骤，就应该这样做。
- **成本受限的任务。** 工作流有有界的步骤数；智能体可能螺旋失控。
- **合规受限的任务。** 审计员想读图，而不是从轨迹中推断。

### 智能体胜过工作流的场景

- **开放性研究。** 当下一步取决于上一步返回的内容时。
- **可变长度任务。** 步骤数未知的数分钟到数小时的工作。
- **新颖领域。** 当你还不知道正确的工作流时——先探索，后规范化。

### 上下文工程的配套

"AI 智能体的有效上下文工程"（Anthropic 2025）将相邻的学科形式化：20 万 token 的窗口是预算，不是容器。什么内容要包含、何时压缩、何时让上下文增长。

## 动手实践

`code/main.py` 针对 `ScriptedLLM` 实现了所有五种工作流模式：

- `prompt_chain(input, steps)`——顺序执行。
- `route(input, classifier, handlers)`——分类 + 分发。
- `parallel_vote(prompt, n, aggregator)`——N 次运行，汇聚。
- `orchestrator_workers(task, workers)`——编排器选择工作器。
- `evaluator_optimizer(task, proposer, evaluator, max_iter)`——循环直到通过。

运行：

```
python3 code/main.py
```

每种模式打印其轨迹。每种模式的代码行数约为 10-15 行；框架的代价是数千行。

## 使用建议

- 大多数任务直接使用 API 调用。
- 只有当模式真正需要持久化状态（LangGraph）、Actor 模型并发（AutoGen v0.4）或角色模板（CrewAI）时才使用框架。
- 当你想要 Claude Code 框架形态而不想重新构建时，使用 Claude Agent SDK。

## 产出技能

`outputs/skill-workflow-picker.md` 为给定的任务描述选择正确的模式，包括决策依据和工作流不够时重构为智能体的路径。

## 练习

1. 用置信度阈值实现路由。阈值以下 -> 升级到人工处理。对于一级支持用例，阈值应该设在哪里？
2. 向 `parallel_vote` 添加超时。当一个调用挂起时会发生什么？如何在缺少投票时汇聚？
3. 将 `evaluator_optimizer` 变成一个 bandit（赌徒算法）：跨迭代保留前 2 个输出，使晚期的好结果不会被晚期的坏结果覆盖。
4. 将提示词链与路由结合：路由器选择三条链之一。与单一大提示词替代方案相比，测量 token 成本。
5. 选择你的一个生产功能。画出工作流图。数步骤数。智能体在这里实际上会更好吗？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 工作流（Workflow） | "预定义流" | 工程师拥有的 LLM 和工具调用图 |
| 智能体（Agent） | "自主 AI" | 模型拥有的图；动态工具指导 |
| 增强型 LLM | "带工具的 LLM" | LLM + 搜索 + 工具 + 记忆；原子单元 |
| 提示词链 | "顺序调用" | 第 N 次调用的输出是第 N+1 次调用的输入 |
| 路由 | "分类器分发" | 选择哪个链/模型处理输入 |
| 并行化 | "扇出" | N 个并发调用；通过分段或投票汇聚 |
| 编排器-工作器 | "调度器智能体" | 编排器 LLM 动态选择专业 LLM |
| 评估器-优化器 | "提议者 + 评判者" | 迭代直到评估器通过；Self-Refine 的推广 |
