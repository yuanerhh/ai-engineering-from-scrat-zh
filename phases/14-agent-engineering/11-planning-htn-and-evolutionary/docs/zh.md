# 使用 HTN 和进化搜索的规划

> 符号规划处理计划可证明正确的情况。进化代码搜索处理适应度函数可被机器检验的情况。ChatHTN（2025）和 AlphaEvolve（2025）展示了与 LLM 配对时各自能解锁什么。

**类型：** 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 02（ReWOO 与计划-执行）
**预计时间：** 约 75 分钟

## 学习目标

- 解释层次任务网络（HTN）：任务、方法、运算符、前置条件、效果。
- 描述 ChatHTN 的混合循环——带 LLM 后备分解的符号搜索。
- 解释 AlphaEvolve 的进化循环，以及为什么它只在有程序化评估器时有效。
- 用标准库实现玩具 HTN 规划器和玩具进化搜索。

## 问题背景

ReWOO（第 02 课）、Plan-and-Execute 和 ReAct 覆盖了大多数智能体规划。有两种情况它们处理得不好：

1. **可证明正确的计划。** 调度、航路规划、合规工作流——计划必须在构建时就是正确的。偶尔会幻觉出一个步骤的流畅 LLM 计划是不可接受的。
2. **具有机器可检验适应度函数的优化。** 矩阵乘法、调度启发式、编译器 pass——目标不是"正确的计划"而是"最好的计划"。

HTN 规划和 AlphaEvolve 解决了两个不同的问题。两者都将 LLM 用作放大器，而非替代者。

## 核心概念

### 层次任务网络（HTN）

HTN 由以下部分组成：

- **任务（Tasks）**——复合任务（待分解）和原始任务（直接可执行）。
- **方法（Methods）**——将复合任务分解为子任务的方式，带前置条件。
- **运算符（Operators）**——带前置条件和效果的原始行动。
- **状态（State）**——一组事实。

规划：给定一个目标任务和初始状态，找到一个将原始运算符分解的序列，使它们的前置条件依次满足。

HTN 比 LLM 出现得更早，至今仍是可证明正确计划的参考标准。

### ChatHTN（Gopalakrishnan 等人，2025）

ChatHTN（arXiv:2505.11814）将符号 HTN 与 LLM 查询交织：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有适用的方法，询问 LLM："在状态 `s` 中你会如何分解 `task`？"
3. 将 LLM 响应转换为候选子任务。
4. 根据运算符 schema 验证；拒绝无效的分解。
5. 递归。

论文的核心声明：生成的每个计划都是可证明正确的，因为 LLM 的建议只作为候选分解进入，而不是直接的计划编辑。符号层拥有正确性；LLM 扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年后续）添加了一个学习器，通过回归泛化 LLM 生成的分解——将 LLM 查询频率降低高达 75%。

### AlphaEvolve（Novikov 等人，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）是另一种不同的方式：由 Gemini 2.0 Flash/Pro 集成体编排的进化代码搜索。

循环：

1. 从一个种子程序 + 程序化评估器（返回适应度分数）开始。
2. LLM 集成体提出变异。
3. 通过评估器运行变异。
4. 保留最好的；再次变异。

已发布的成果：

- 56 年来首次改进 Strassen 用于 4×4 复数矩阵乘法（48 次标量乘法）。
- 通过 Borg 调度启发式恢复了 0.7% 的 Google 计算资源。
- 在前沿工作负载上 FlashAttention 加速 32%。

硬约束：适应度函数必须可被机器检验。对散文答案进行进化搜索不会收敛。

### 何时使用哪种

| 问题类别 | 使用方案 | 原因 |
|---------|---------|------|
| 带硬约束的调度 | HTN + ChatHTN | 可证明的正确性 |
| 编译器优化 | AlphaEvolve | 机器可检验的适应度 |
| 多步骤任务执行 | ReAct / ReWOO | LLM 在循环中，无正式保证 |
| 带测试的代码改进 | AlphaEvolve | 测试就是评估器 |
| 策略绑定的自动化 | HTN | 前置条件编码策略 |

### 这个模式在哪里出错

- **没有运算符的 HTN。** 没有前置条件/效果 schema，正确性声明就崩溃了。ChatHTN 的"LLM 建议分解"需要 schema 来拒绝无效行动。
- **没有真实评估器的 AlphaEvolve。** "问 LLM 代码是否更好"不是适应度函数。评估器必须是确定性的且快速的。
- **过度工程化。** 大多数智能体任务不需要这两种。先用 ReAct 或 ReWOO。

## 动手实践

`code/main.py` 实现了两个玩具：

- 一个标准库 HTN 规划器，带有运算符、方法、前置条件、效果，以及一个在没有方法匹配复合任务时启动的 `LLMFallback`。"LLM"是一个脚本化的分解器，使规划器可以离线运行。
- 一个对算术程序进行标准库进化搜索：生长使 `|f(x) - target|` 在测试集上最小化的表达式。评估器是确定性的。

运行：

```
python3 code/main.py
```

轨迹显示 HTN 规划器分解复合任务（带计划中途的 LLM 后备），以及进化循环收敛到目标表达式。

## 使用建议

- **HTN 规划器**——`pyhop`、`SHOP3`，或为特定领域的策略执行构建你自己的。
- **ChatHTN**——研究代码；该模式（符号 + LLM 后备）可以简洁地移植到任何 HTN 规划器。
- **AlphaEvolve**——DeepMind 论文；该模式（集成 + 评估器）是可复现的。OpenEvolve 和类似的开源分支正在涌现。
- **智能体框架**——目前没有提供一流的 HTN 或 AlphaEvolve 支持。将其构建为子智能体或后台工作器。

## 产出技能

`outputs/skill-hybrid-planner.md` 生成一个混合规划器脚手架（HTN 或进化），LLM 角色明确地被界定范围。

## 练习

1. 用回溯扩展 HTN 规划器：当运算符的后置条件在运行时失败时，回滚并尝试下一个方法。
2. 向 ChatHTN 添加 LLM 方法缓存：当 LLM 在状态模式 `P` 下分解任务 `T` 时，存储结果。下次调用时先检查方法库。
3. 将进化搜索评估器替换为真实的测试套件。进化一个通过 20 个测试用例的排序函数；报告收敛所需的世代数。
4. 阅读 AlphaEvolve 的评估器设计说明。为你关心的领域设计一个评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 组合两者：使用 HTN 将复合任务分解为子任务，然后对每个子任务的原始运算符使用进化搜索。它在哪里大放异彩，在哪里过度工程化？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| HTN | "层次规划器" | 带运算符、前置条件、效果的任务分解 |
| 方法（Method） | "分解规则" | 将复合任务分解为子任务的方式 |
| 运算符（Operator） | "原始行动" | 带前置条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 符号规划器在没有方法匹配时询问 LLM |
| AlphaEvolve | "进化代码搜索" | 集成 LLM 变异代码；确定性评估器选择 |
| 适应度函数 | "评估器" | 对输出进行确定性、机器可检验评分 |
| 在线方法学习 | "缓存的 LLM 分解" | 存储 + 泛化 LLM 计划以降低查询成本 |
