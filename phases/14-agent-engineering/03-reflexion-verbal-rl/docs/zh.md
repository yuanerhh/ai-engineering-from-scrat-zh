# Reflexion：语言强化学习

> 基于梯度的强化学习需要数千次试验和一个 GPU 集群来修复一个失败模式。Reflexion（Shinn 等人，NeurIPS 2023）用自然语言做到了这一点：每次失败的试验后，智能体写一篇反思，存入情节记忆，并在下一次试验中以该记忆为条件。这是 Letta 的睡眠时间计算、Claude Code 的 CLAUDE.md 学习记录和 pro-workflow 的 learn-rule 背后的模式。

**类型：** 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 14 · 01（智能体循环）、Phase 14 · 02（ReWOO）
**预计时间：** 约 60 分钟

## 学习目标

- 说出 Reflexion 的三个组件（Actor、Evaluator、Self-Reflector）和情节记忆的作用。
- 用标准库实现一个带二元评估器、反思缓冲区和全新重试的 Reflexion 循环。
- 针对给定任务在标量、启发式和自我评估的反馈来源之间做出选择。
- 解释为什么语言强化能捕获梯度强化学习需要数千次试验才能修复的错误。

## 问题背景

一个智能体失败了。在标准强化学习中，你需要运行数千次额外试验、计算梯度、更新权重。这既昂贵又缓慢，而且大多数生产智能体对每次失败都没有训练预算。

Reflexion（Shinn 等人，arXiv:2303.11366）提出了一个不同的问题：如果智能体只是思考为什么失败，并带着这个思考重试呢？不需要权重更新，不需要梯度，只需要在试验之间存储的自然语言。

结果：在 ALFWorld 上它击败了 ReAct 和其他非微调基准。在 HotpotQA 上它优于 ReAct。在代码生成（HumanEval/MBPP）上它创造了当时的最先进水平。这一切都没有任何梯度步骤。

## 核心概念

### 三个组件

```
Actor（行动器）:         生成一条轨迹（ReAct 风格的循环）
Evaluator（评估器）:     对轨迹评分——二元、启发式或自我评估
Self-Reflector（自我反思器）: 对失败写一段自然语言反思
```

加上一个数据结构：

```
Episodic memory（情节记忆）: 先前反思的列表，在下一次试验的提示词中前置
```

一次试验运行 Actor。Evaluator 对其评分。如果分数低，Self-Reflector 生成一段反思（"我因为将问题误读为询问 X 而选择了错误的工具，但实际上是在询问 Y"）。反思进入情节记忆。下一次试验重新开始但能看到这段反思。

### 三种评估器类型

1. **标量（Scalar）**——外部二元信号。ALFWorld 成功或失败。HumanEval 测试通过或失败。最简单，信号最强。
2. **启发式（Heuristic）**——预定义的失败特征。"如果智能体连续两次产生相同的行动，标记为卡住。""如果轨迹超过 50 步，标记为低效。"
3. **自我评估（Self-evaluated）**——LLM 对自己的轨迹评分。在没有真实标签时需要。信号较弱；与工具驱动的验证配合使用效果好（第 05 课——CRITIC）。

2026 年的默认做法是混合使用：有标签时用标量，没有时用自我评估，启发式作为安全护栏。

### 为什么这具有普适性

Reflexion 与其说是一种新算法，不如说是一种有命名的模式。几乎每一个生产"自愈"智能体都运行某种变体：

- Letta 的睡眠时间计算（第 08 课）：一个独立智能体反思过去的对话并写入记忆块。
- Claude Code 的 `CLAUDE.md` / "保存记忆"模式：反思作为学习记录被捕获，在未来会话中前置。
- pro-workflow 的 `/learn-rule` 命令：纠正被捕获为显式规则。
- LangGraph 的反思节点：一个对输出评分的节点，如果需要则路由到精炼。

所有这些都来自同一个洞察：自然语言是一种足够丰富的媒介，可以在运行之间传递"我从失败中学到了什么"。

### 何时有效，何时无效

Reflexion 在以下情况有效：

- 有明确的失败信号（测试失败、工具错误、答案错误）。
- 任务类别可重现（可以再次提同类问题）。
- 反思有空间改进轨迹（足够的行动预算）。

Reflexion 在以下情况无效：

- 智能体第一次就成功了。
- 失败是外部的（网络中断、工具损坏）——"网络中断了"的反思对未来的运行没有帮助。
- 反思变成迷信——存储了关于一次性不稳定运行的叙述。

2026 年的陷阱：记忆腐化。反思积累；一些已过时或错误；随着情节缓冲区增长，重新运行变得越来越慢。缓解措施：定期压缩（第 06 课）、反思的 TTL（生存时间），或独立的睡眠时间清理智能体（Letta）。

## 动手实践

`code/main.py` 在一个玩具谜题上实现了 Reflexion：生成一个总和等于目标的 3 元素列表。Actor 输出候选列表；Evaluator 检查总和；Self-Reflector 写一行关于出了什么问题的说明。反思进入情节记忆供下一次试验使用。

组件：

- `Actor`——一个看到反思后会改进的脚本化策略。
- `Evaluator.binary()`——目标总和的通过/失败。
- `SelfReflector`——生成失败的一行诊断。
- `EpisodicMemory`——一个带 TTL 语义的有界列表。

运行：

```
python3 code/main.py
```

轨迹显示三次试验。第 1 次失败，存储一条反思，第 2 次看到反思并有所改进但仍然失败，第 3 次成功。与基准运行（无反思）比较——基准停留在第 1 次的答案上。

## 使用建议

LangGraph 将反思作为节点模式提供。Claude Code 的 `/memory` 命令和 pro-workflow 的 `/learn-rule` 将情节缓冲区外化为 markdown 文件。Letta 的睡眠时间计算在空闲时运行 Self-Reflector，使主智能体保持延迟敏感性。OpenAI Agents SDK 不直接提供 Reflexion；你通过自定义 Guardrail（按分数拒绝轨迹）和在运行之间存活的记忆 `Session` 来构建它。

## 产出技能

`outputs/skill-reflexion-buffer.md` 创建并维护一个带反思捕获、TTL 和去重的情节缓冲区。给定一个任务类别和一次失败，它输出一条真正能帮助下一次试验的反思（而不是笼统的"要更加小心"）。

## 练习

1. 从二元评估器切换到返回距离指标（距目标有多远）的标量评估器。它收敛得更快吗？
2. 为反思添加 10 次试验的 TTL。超过这个点后，旧反思是有帮助还是有害？
3. 实现启发式评估器：如果同一行动重复出现则标记试验为卡住。这与 Self-Reflector 如何交互？
4. 运行带有忽略反思的对抗性 Actor 的 Reflexion。迫使 Actor 注意到反思所需的最小反思提示词工程是什么？
5. 阅读 Reflexion 论文第 4 节关于 AlfWorld 的内容。从概念上复现 130% 成功率提升：相比普通 ReAct 的关键差异是什么？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Reflexion | "自我纠错" | Shinn 等人 2023——Actor、Evaluator、Self-Reflector 加情节记忆 |
| 语言强化 | "无梯度学习" | 在下一次试验的提示词中前置的自然语言反思 |
| 情节记忆（Episodic memory） | "每任务反思" | 一个任务类别的先前反思的有界缓冲区 |
| 标量评估器 | "二元成功信号" | 来自真实标签的通过/失败或数值分数 |
| 启发式评估器 | "基于模式的检测器" | 预定义的失败特征（如卡住循环、步骤过多） |
| 自我评估器 | "LLM 评判自身轨迹" | 当没有真实标签时的低信号后备——与工具驱动验证配合 |
| 记忆腐化 | "陈旧反思" | 情节缓冲区充满过时条目；通过压缩/TTL 修复 |
| 睡眠时间反思 | "异步自我反思" | 在非关键路径上运行 Self-Reflector 使主智能体保持速度 |
