# 工具接口——为何智能体需要结构化输入输出

> 语言模型产生词元，程序执行动作。填补这两者之间差距的就是工具接口：一份允许模型请求动作、由宿主执行该动作的契约。2026 年的每个技术栈——OpenAI、Anthropic 和 Gemini 的函数调用；MCP 的 `tools/call`；A2A 的任务部件——都是同一个四步循环的不同编码方式。本课给这个循环命名，并展示运行它所需的最少机制。

**类型：** 学习
**编程语言：** Python（标准库，无需 LLM）
**前置知识：** Phase 11（LLM 完成 API）
**预计时间：** 约 45 分钟

## 学习目标

- 解释为何只能生成文本的 LLM 无法自行对真实世界执行动作。
- 画出四步工具调用循环（描述 → 决策 → 执行 → 观察），并说明每步的所有者。
- 将工具描述写成三个部分：名称、JSON Schema 输入和确定性执行器函数。
- 区分纯工具和有副作用工具，阐述这种区分对安全性的重要性。

## 问题背景

LLM 输出的是下一个词元的概率分布——这就是其全部输出接口。如果你问聊天模型"班加罗尔现在的天气是什么"，它可以写出一句听起来合理的话，但它无法真正连接到天气 API。这句话可能碰巧是对的，也可能已经过时三天了。

工具接口的目的就是填补这个差距。宿主程序——你的智能体运行时、Claude Desktop、ChatGPT、Cursor 或自定义脚本——向模型公告一个可调用的工具列表。当模型判断需要执行某个动作时，它会输出一个结构化的载荷，指明要调用的工具及其参数。宿主解析这个载荷，真正运行工具，并将结果反馈回来。这个循环持续到模型判断不再需要更多调用为止。

这份契约的第一个版本于 2023 年 6 月作为 OpenAI 的 "functions" 参数发布。Anthropic 在 Claude 2.1 中用 `tool_use` 块跟进，Gemini 几个月后添加了 `functionDeclarations`。现在每家提供商都暴露了相同的形式：入参是 JSON Schema 类型化的工具列表，出参是 JSON 载荷的工具调用。模型上下文协议（Model Context Protocol，2024 年 11 月）将这份契约通用化，使得一个工具注册表可以服务于所有模型。A2A（2026 年 4 月，v1.0）在此基础上为智能体间委托添加了同样的原语。

四步循环是所有这些的不变量，其余一切都是对它的细化。

## 核心概念

### 第一步：描述

宿主用三个字段声明每个工具：

- **名称。** 稳定的、机器可读的标识符。用 `get_weather`，不用"天气功能"。
- **描述。** 一段自然语言说明。"在用户询问某个特定城市的当前天气状况时使用。不用于历史数据。"
- **输入 Schema。** 描述工具参数的 JSON Schema 对象（草案 2020-12）。

模型接收这个列表。现代提供商使用提供商特定的模板将这些声明序列化到系统提示中，所以调用方只需处理结构化形式。

### 第二步：决策

给定用户消息和可用工具，模型从三种行为中选择一种：

1. **直接用文本回答。** 不调用工具。
2. **调用一个或多个工具。** 输出结构化调用对象。在 `parallel_tool_calls: true` 下（OpenAI 和 Gemini 默认开启，Anthropic 需显式开启），模型可以在一轮中输出多个调用。
3. **拒绝。** 严格模式的结构化输出可以产生类型化的 `refusal` 块而非调用。

工具调用载荷有三个稳定字段：调用的 `id`、工具 `name` 和 JSON `arguments` 对象。`id` 的存在使宿主能够将后续结果与特定调用关联起来，这在并行调用乱序返回时尤为重要。

### 第三步：执行

宿主接收调用，根据声明的 schema 验证参数，然后运行执行器。参数无效意味着模型幻觉了某个字段或使用了错误的类型——这在弱模型上是非常常见的失败模式。生产宿主对无效参数有三种处理方式之一：快速失败并将错误暴露给模型、用约束解析器修复 JSON，或者在提示中包含验证错误后重试模型。

执行器本身是普通代码——Python、TypeScript、Shell 命令、数据库查询。它产生一个结果，通常是字符串，但也可以是任何 JSON 值或结构化内容块（MCP 中的文本、图像或资源引用）。结果必须是可序列化的。

### 第四步：观察

宿主将工具结果追加到对话中（作为带匹配 `id` 的 `tool` 角色消息），然后重新调用模型。模型现在可以在上下文中看到工具输出，可以产生最终答案或请求更多调用。这个过程持续到模型停止输出调用或宿主达到迭代次数安全限制为止。

### 信任分割

工具分为两种在安全性上很重要的类型：

- **纯工具。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以安全地投机性调用。
- **重要后果工具。** 改变状态、花费金钱、接触用户数据。`send_email`、`delete_file`、`execute_trade`。必须设置门控。

Meta 2026 年的智能体安全"二规则"说：单次轮次最多可以组合以下两项：不可信输入、敏感数据、有重要后果的动作。工具接口就是执行这条规则的地方——通过拒绝调用、要求用户确认或升级范围。完整的安全章节见 Phase 13 · 15，智能体级权限策略见 Phase 14 · 09。

### 循环所在

| 场景 | 谁描述 | 谁决策 | 谁执行 |
|------|--------|--------|--------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | LLM（通过 MCP 客户端） | MCP 服务器 |
| A2A | 智能体卡发布者 | 调用智能体 | 被调用智能体 |
| Web 浏览器（函数调用智能体） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

所有场景都是同样的四步。列名会变化，结构不变。

### 为何不直接提示模型输出 JSON？

"让模型以 JSON 格式回复"是函数调用出现前的模式，在前沿模型上约有 5%-15% 的失败率，在较小的模型上更高。失败模式包括缺少大括号、尾随逗号、幻觉字段和错误类型，之后需要 JSON 修复、重试或约束解码器。

原生函数调用有三方面优势：第一，提供商端到端地在精确的调用形式上训练模型，严格模式下有效 JSON 率提升到 98%-99%。第二，调用载荷位于其专用协议槽中，而非自由文本内——工具调用不会泄漏到用户可见的回复中。第三，提供商通过约束解码（OpenAI 的严格模式、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）强制执行 schema 合规，输出保证验证通过。

### 断路器

当模型停止输出调用或宿主达到最大轮次数时，循环终止。生产宿主通常将此设置为 5 到 20 轮。超过这个数字，几乎可以肯定是模型无法跳出的循环。Claude Code 默认为 20，OpenAI Assistants 默认为 10，Cursor 智能体模式默认为 25。

另一种方案——无界循环——每隔半年就会出现"智能体一夜之间花费了 400 美元 API 费用"的事故报告。不要在没有上限的情况下上线。

## 动手实践

`code/main.py` 在不使用 LLM 的情况下运行四步循环。一个假的"决策者"函数通过对用户消息进行模式匹配来模拟模型；执行器、schema 验证器和观察步骤的框架是真实的。运行它查看包含可打印中间状态的完整请求/响应编排，然后在后续课程中将假决策者替换为任何真实提供商。

重点关注：
- 工具注册表为每个工具保存三个字段：名称、描述、schema 和执行器引用。
- 验证器是仅用标准库编写的最小 JSON Schema 子集（类型、必填、枚举、最小/最大值）。Phase 13 · 04 会有更完整的版本。
- 循环将迭代次数限制在 5 次。生产智能体正需要这种断路器。

## 产出技能

本课产出 `outputs/skill-tool-interface-reviewer.md`：给定一份草拟的工具定义（名称 + 描述 + schema + 执行器概要），该技能审核其循环适用性：名称是否机器稳定，描述是否是完整的使用说明，schema 是否正确使用 JSON Schema 2020-12，纯工具与有副作用工具的分类是否明确。

## 练习

1. 在 `code/main.py` 中添加第四个工具 `get_stock_price(ticker)`。描述写为"当用户按股票代号询问当前股价时使用。不用于历史价格或市场摘要。"运行框架，确认假决策者将提及股票代号的查询路由到新工具。

2. 破坏 schema 验证器。传入一个 `arguments` 对象缺少必填字段的调用，确认宿主在执行前拒绝它。再传入一个包含额外未知字段的调用，决策：宿主应该拒绝还是忽略？用安全论点为你的选择辩护。

3. 将框架中的每个工具分类为纯工具或有副作用工具。为需要的注册表条目添加 `consequential: true` 标志，并修改循环，在选择有副作用工具时打印"将向用户确认"的提示。这就是每个生产宿主都需要的确认门控的雏形。

4. 在纸上画出四步循环，并填入上表中你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT 或自定义技术栈）对应的列。与 Phase 13 · 06 中的 MCP 特定变体交叉对照。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 工具（Tool） | "模型可以调用的东西" | 名称 + JSON Schema 类型化输入 + 执行器函数的三元组 |
| 函数调用（Function calling） | "原生工具使用" | 提供商级 API 支持，用于输出结构化工具调用而非散文 |
| 工具调用（Tool call） | "模型的行动请求" | 模型输出的包含 `id`、`name`、`arguments` 的 JSON 载荷 |
| 工具结果（Tool result） | "工具返回的内容" | 执行器的输出，包装在带匹配 id 的 `tool` 角色消息中 |
| 并行工具调用 | "同时多个调用" | 一个模型轮次中的多个调用对象，独立且可按 id 排序 |
| 严格模式（Strict mode） | "保证 JSON" | 强制模型输出符合声明 schema 的约束解码 |
| 纯工具（Pure tool） | "只读工具" | 无副作用；可安全重新运行 |
| 有副作用工具（Consequential tool） | "动作工具" | 改变外部状态；需要门控、审计或用户确认 |
| 四步循环 | "工具调用循环" | 描述 → 决策 → 执行 → 观察 |
| 宿主（Host） | "智能体运行时" | 持有工具注册表、调用模型并运行执行器的程序 |
