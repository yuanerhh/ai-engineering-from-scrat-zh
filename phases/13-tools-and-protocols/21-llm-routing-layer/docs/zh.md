# LLM 路由层——LiteLLM、OpenRouter、Portkey

> 厂商锁定代价高昂。不同的工具调用工作负载适合不同的模型。路由网关提供统一的 API 接口、重试、故障转移、成本追踪和防护栏。2026 年有三种主要类型：LiteLLM（开源自托管）、OpenRouter（托管 SaaS）、Portkey（生产级，2026 年 3 月开源）。本课列出决策标准，并讲解一个标准库路由网关的实现。

**类型：** 学习
**编程语言：** Python（标准库，路由 + 故障转移 + 成本追踪器）
**前置知识：** Phase 13 · 02（函数调用）、Phase 13 · 17（网关）
**预计时间：** 约 45 分钟

## 学习目标

- 区分自托管、托管和生产级路由选项。
- 实现一个在提供商失败时按定义优先级重试的故障转移链。
- 跨提供商追踪每请求的成本和令牌使用情况。
- 针对给定的生产约束，在 LiteLLM、OpenRouter 和 Portkey 之间做出选择。

## 问题背景

提供商路由重要的场景：

1. **成本。** Claude Sonnet 的成本是 Haiku 的 3 倍。对于分类任务，Haiku 已经足够；对于综合任务，Sonnet 物有所值。按请求路由。

2. **故障转移。** OpenAI 某小时状态不佳，每个请求都失败。你希望在不重新部署的情况下自动回退到 Anthropic。

3. **延迟。** 实时聊天 UI 需要快速的首令牌时间。批量摘要器不需要。按延迟 SLA 路由。

4. **合规。** 欧盟用户必须留在欧盟区域。按地区路由。

5. **实验。** 对同一工作负载进行两个模型的 A/B 测试。按测试桶路由。

为每个集成手动编码所有这些是重复工作。路由网关提供一个兼容 OpenAI 的 API，处理其余一切。

## 核心概念

### OpenAI 兼容代理形态

所有人都使用 OpenAI 形态。路由网关暴露 `/v1/chat/completions`，接受 OpenAI schema，内部代理到 Anthropic / Gemini / Cohere / Ollama / 任何东西。客户端不需要关心。

### 模型别名

你的代码不写 `claude-3-5-sonnet-20251022`，而是写 `our_smart_model`。网关将别名映射到真实模型。当 Anthropic 发布 Claude 4 时，你在服务器端更改别名；你的代码无需任何修改。

### 故障转移链

```
primary: openai/gpt-4o
on 5xx: anthropic/claude-3-5-sonnet
on 5xx: google/gemini-1.5-pro
on 5xx: refuse
```

网关在配置中定义这一逻辑。重试计入预算，防止故障转移级联导致成本爆炸。

### 语义缓存

相同或近似相同的提示词命中缓存而非提供商。重复智能体循环的节省可达 30% 至 60%。键基于嵌入；近似相同的提示词共享一个缓存槽。

### 防护栏

网关层面：

- **PII 编辑。** 在发送提示词之前进行基于正则表达式或机器学习的处理。
- **策略违规。** 拒绝包含禁止内容的提示词。
- **输出过滤器。** 清洗补全结果中的泄露内容。

Portkey 和 Kong 都提供了有主见的防护栏。LiteLLM 将其设为可选。

### 每密钥速率限制

一个 API 密钥 = 一个团队。每密钥预算防止一个团队耗尽共享配额。大多数网关支持这一功能。

### 自托管 vs 托管的权衡

| 因素 | LiteLLM（自托管） | OpenRouter（托管） | Portkey（生产级） |
|------|----------------|-----------------|----------------|
| 代码 | 开源，Python | 托管 SaaS | 开源（2026 年 3 月）+ 托管 |
| 设置 | 部署一个代理 | 注册账号 | 两者均可 |
| 提供商 | 100+ | 300+ | 100+ |
| 计费 | 使用你自己的密钥 | OpenRouter 积分 | 使用你自己的密钥 |
| 可观测性 | OpenTelemetry | 仪表盘 | 完整 OTel + PII 编辑 |
| 最适合 | 需要完全控制的团队 | 快速原型开发 | 需要合规保证的生产环境 |

当你有 SRE 团队且需要数据主权时，选择 LiteLLM。当你需要单一订阅且不想维护基础设施时，选择 OpenRouter。当你需要开箱即用的防护栏和合规功能时，选择 Portkey。

### 成本追踪

每个请求携带 `provider`、`model`、`input_tokens`、`output_tokens`。乘以网关维护的每模型每令牌定价表中的价格。按用户/团队/项目聚合。

### MCP 加路由

网关可以同时路由 LLM 调用和 MCP 采样请求。当采样请求的 modelPreferences 偏好特定模型时，网关会翻译到正确的后端。这就是 Phase 13 · 17（MCP 网关）和本课路由网关有时合并为一个服务的地方。

### 路由策略

- **静态优先级。** 列表中的第一个；出错时回退。
- **负载均衡。** 轮询或加权。
- **成本感知。** 选择满足延迟/质量的最便宜模型。
- **延迟感知。** 选择过去 N 分钟内最快的模型。
- **任务感知。** 提示词分类器将编码路由到某个模型，将摘要路由到另一个模型。

## 动手实践

`code/main.py` 以约 150 行代码实现了一个路由网关：接受 OpenAI 格式的请求，翻译到每个提供商的存根，运行优先级故障转移链，追踪每请求的成本，并对输入应用 PII 编辑处理。运行三种场景：正常请求、主要提供商故障触发回退、PII 泄露被编辑捕获。

重点关注：
- `ROUTES` 字典：别名 -> 按优先级排序的具体提供商列表。
- 故障转移循环在 5xx 时重试。
- 成本追踪器将令牌使用量乘以每模型费率。
- PII 编辑器在转发前清洗 SSN 格式的模式。

## 练习

1. 运行 `code/main.py`。触发故障场景；确认故障转移落在第二个提供商上，成本归因正确。

2. 添加语义缓存：提示词的 SHA256 作为查找键；缓存命中立即返回。测量重复调用的成本节省。

3. 添加一个提示词分类器，将"代码 ..."提示词路由到倾向智能的别名，将"总结 ..."提示词路由到倾向速度的别名。

4. 设计每团队预算：每个团队有每月支出上限；达到上限时网关拒绝请求。选择执行粒度（每请求或窗口化）。

5. 并排阅读 LiteLLM、OpenRouter 和 Portkey 的文档。各找出其中一个另外两者没有的功能。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 路由网关 | "LLM 代理" | 位于多个提供商前面的统一 API 接口层 |
| OpenAI 兼容 | "使用 OpenAI schema" | 接受 `/v1/chat/completions` 格式，翻译到任何后端 |
| 模型别名 | "our_smart_model" | 代码中的名称，网关将其映射到具体模型 |
| 故障转移链 | "重试列表" | 失败时按顺序尝试的提供商列表 |
| 语义缓存 | "提示词嵌入缓存" | 以提示词的嵌入为键；近似重复共享缓存命中 |
| 防护栏 | "输入/输出过滤器" | 编辑 PII，拒绝策略违规 |
| 每密钥速率限制 | "团队预算" | 绑定到 API 密钥的配额 |
| 成本追踪 | "每请求支出" | 令牌使用量 × 每模型价格的聚合 |
| LiteLLM | "开源代理" | 可自托管的开源路由网关 |
| OpenRouter | "托管 SaaS" | 基于积分计费的托管网关 |
| Portkey | "生产级选项" | 内置防护栏的开源 + 托管方案 |
