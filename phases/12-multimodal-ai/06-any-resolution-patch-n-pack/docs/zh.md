# 任意分辨率视觉：Patch-n'-Pack 与 NaFlex

> 真实图像不是 224×224 的正方形。收据是 9:16，图表是 16:9，医学扫描可能是 4096×4096，手机截图是 9:19.5。2024 年之前 VLM 的做法——将所有图像缩放到固定正方形——丢弃了使 OCR、文档理解和高分辨率场景解析得以工作的信号。NaViT（Google，2023）证明可以将可变分辨率的块通过块对角掩码打包到一个 Transformer 批次中。Qwen2-VL 的 M-RoPE（2024）完全抛弃了绝对位置表。LLaVA-NeXT 的 AnyRes 将高分辨率图像分割为基础图像加子图像的形式。SigLIP 2 的 NaFlex 变体（2025）现已成为希望用单一检查点处理所有长宽比的开放 VLM 的默认编码器。本课端到端实现 patch-n'-pack。

**类型：** 构建
**编程语言：** Python（标准库，图像块打包器 + 块对角掩码）
**前置知识：** Phase 12 · 01（ViT 图像块）、Phase 12 · 05（LLaVA）
**预计时间：** 约 120 分钟

## 学习目标

- 将一批不同分辨率图像的块打包成一个序列，并构建块对角注意力掩码。
- 针对给定任务在 AnyRes 拼接（LLaVA-NeXT）、NaFlex（SigLIP 2）和 M-RoPE（Qwen2-VL）之间做出合理选择。
- 在不缩放图像的情况下，计算 OCR、图表和摄影场景的词元预算。
- 说出正方形缩放的三种失败模式：文字压缩变形、内容裁剪丢失、填充占用大量词元。

## 问题背景

Transformer 处理的是序列，批次是相同长度序列的堆叠。如果图像是 224×224，每次得到 196 个块词元，无需填充，问题解决。在 224 上训练，在 224 上推理，永远不用再考虑分辨率。

但现实世界不会配合。文档是纵向的（8.5×11 英寸，约 2:3）；图表截图是横向的（16:9）；收据又高又细（1:3）；医学图像的分辨率是 2048×2048 或更大；手机截图是 1170×2532（0.46:1）。

2024 年之前的三种方案及其失败原因：

1. 缩放到固定正方形（224×224 或 336×336）：压缩变形会扭曲文字和人脸，缩小图像会破坏图表标签和 OCR 内容。LLaVA-1.5 之前的标准做法。
2. 裁剪到固定长宽比：丢弃了大部分图像，且选择裁剪位置本身就是一个视觉问题。
3. 填充到最长边：解决了变形问题，但对纵向图像会有 50%+ 的词元浪费在填充上。所有填充词元的注意力计算代价是二次方的。

2024–2025 年的答案：让 Transformer 以图像的原生分辨率处理块，并设计出将异构批次打包成一个序列同时避免浪费计算的方法。

## 核心概念

### NaViT 与 patch-n'-pack

NaViT（Dehghani 等，2023）是证明该方法在规模上有效的论文。思路非常直接：

1. 对批次中的每张图像，以选定的块大小（如 14）计算其原生块网格。
2. 将每张图像的块展平为各自的可变长度序列。
3. 将所有图像的块拼接成一个批次的长序列。
4. 构建块对角注意力掩码，使图像 A 的块只在图像 A 内部进行注意力计算。
5. 携带每个块的位置信息（2D RoPE 或分数位置嵌入）。

三张分辨率分别为 336×336（576 词元）、224×224（256 词元）和 448×336（768 词元）的图像组成的批次，变为一个 1600 词元的序列，带有 1600×1600 的块对角掩码。无填充，无浪费计算，Transformer 处理任意长宽比。

NaViT 还在训练时引入了分数块丢弃——在批次中随机丢弃 50% 的块——这既起到了正则化的作用，又加速了训练。SigLIP 2 继承了这一做法。

### AnyRes（LLaVA-NeXT）

LLaVA-NeXT 的 AnyRes 是一种实用的替代方案。给定一张高分辨率图像和一个固定编码器（336 分辨率的 CLIP 或 SigLIP），对图像进行分块拼接：

1. 从预定义的网格布局集合中选择最匹配图像长宽比的方案——(1×1)、(1×2)、(2×1)、(1×3)、(3×1)、(2×2) 等。
2. 将完整图像分割到网格中，每个格变成一个 336×336 的裁剪块。
3. 还生成一个缩略图：将整张图像缩放到 336×336 作为全局上下文词元。
4. 通过冻结的 336-编码器对每个格编码，拼接所有格的词元 + 缩略图词元。

对于一张 672×672 图像使用 2×2 网格加缩略图：4 × 576 + 576 = 2880 个视觉词元。成本高但有效——LLM 同时看到了局部细节和全局上下文。

当编码器被冻结且只支持单一分辨率时，AnyRes 是首选方案。对于大图像，它会使词元数量爆炸（一张 1344×1344 图像在 4×4 网格下约有 9216 + 576 ≈ 9800 个词元，几乎填满 8k 的 LLM 上下文）。

### M-RoPE（Qwen2-VL）

Qwen2-VL 引入了多模态旋转位置嵌入（M-RoPE）。与 NaViT 的分数位置或 AnyRes 的分块拼接不同，每个块携带一个三维位置（时间、高度、宽度）。查询/键的旋转处理任意 H、W 和时间长度。

M-RoPE 无需重新训练即可支持原生动态分辨率。推理时输入任意 H×W 图像，块嵌入器产生 H/14 × W/14 个词元，每个词元获得其 (t=0, r=行, c=列) 位置，RoPE 以正确的频率旋转注意力，完成。Qwen2.5-VL 和 Qwen3-VL 延续了这一设计，InternVL3 的 V2PE 也是同样的思路，只是对每种模态使用不同的编码。

与 AnyRes 不同，M-RoPE 在原生分辨率下是 O(H × W / P²) 个词元，无需乘法块开销；与 NaViT 不同，它每次前向传播仍处理单张图像，批次跨分辨率时仍需在顶层使用 patch-n'-pack。

### NaFlex（SigLIP 2）

NaFlex 是 SigLIP 2 检查点的原生灵活模式。单个模型在推理时支持多种序列长度（256、729、1024 词元）。内部在训练时使用 NaViT 风格的 patch-n'-pack，每个块使用绝对分数位置。卖点在于：一个检查点，根据任务在推理时选择词元预算。

语义任务（分类、检索）用 256 词元；OCR 或图表理解用 1024 词元，无需重新训练。

### 打包掩码

块对角掩码是大多数实现踩坑的地方。对于总长度 `N_total`、包含 `i=0..B-1` 张图像、各长度为 `n_i` 的打包序列，形状为 `(N_total, N_total)` 的掩码 `M` 在两个索引都落在同一图像块内时为 1，否则为 0。可以用累积长度列表构建：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 当且仅当存在 b 使得 offsets[b] <= i < offsets[b+1] 且 offsets[b] <= j < offsets[b+1]
```

在 PyTorch 中，可以用 `torch.block_diag` 一行实现，或直接 gather。FlashAttention 的变长路径（`cu_seqlens`）完全跳过掩码，直接使用累积长度张量在序列内进行注意力计算——对于典型批次，比密集掩码快约 10 倍。

### 词元预算

按任务选择策略：

- OCR / 文档：1024–4096 词元。SigLIP 2 NaFlex 用 1024，或 AnyRes 3×3 + 缩略图。
- 图表和 UI：在 384–448 原生分辨率下 729–1024 词元。Qwen2.5-VL 动态分辨率，设置最大像素上限。
- 自然照片：256–576 词元已足够，LLM 能看到足够信息，在内容密度高的地方才付出词元代价。
- 视频：空间池化后每帧 64–128 词元，2–8 FPS。12.17 课详细介绍。

2026 年的生产规则：按任务设置每像素上限，在不超过该上限的情况下以原生长宽比编码，打包批次，跳过填充。Qwen2.5-VL 正是为此目的提供了 `min_pixels` 和 `max_pixels` 参数。

## 动手实践

`code/main.py` 对一批整数像素坐标的异构图像实现了 patch-n'-pack，功能包括：

- 接受 (H, W) 图像尺寸列表。
- 计算每张图像在块大小 14 下的序列长度。
- 将它们打包成总长度 `Σn_i` 的一个序列。
- 构建块对角注意力掩码（密集形式，便于理解）。
- 对比打包成本 vs 正方形缩放和 AnyRes 分块拼接。
- 打印混合批次（收据、图表、截图、照片）的词元预算表格。

运行它。输出的数字正是 2026 年每个开放 VLM 都使用 patch-n'-pack 的原因。

## 产出技能

本课产出 `outputs/skill-resolution-budget-planner.md`：给定混合长宽比的工作负载（OCR、图表、照片、视频帧）和总词元预算，选择合适的策略（NaFlex、AnyRes、M-RoPE 或固定正方形）并输出每个请求的配置。在为产品定制 VLM 时使用此技能，可防止悄然发生的 10 倍词元爆炸拖垮延迟预算。

## 练习

1. 收据尺寸为 600×1500（1:2.5）。Patch 14 下原生分辨率有多少词元？缩放到 336 的正方形后有多少？实际上哪种 OCR 准确率损失更多？

2. 为长度分别为 256、576、729、1024 的四张图像构建块对角掩码。验证注意力矩阵为 2585×2585，且恰好有 `256² + 576² + 729² + 1024²` 个非零条目。

3. 对 1792×896 图像（Patch 14），比较：(a) 缩放到 336 再编码，(b) AnyRes 2×1 + 缩略图，(c) M-RoPE 原生分辨率。哪种词元最少？哪种保留细节最多？

4. 实现分数块丢弃：给定打包序列，随机丢弃 50% 的词元，并相应更新块对角掩码，测量掩码稀疏度的变化。

5. 阅读 Qwen2-VL 论文（arXiv:2409.12191）第 3.2 节。用两句话描述 `min_pixels` 和 `max_pixels` 控制什么，以及为何两个边界都很重要。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Patch-n'-pack | "NaViT 风格打包" | 将不同图像的可变长度块序列拼接到一个批次维度中 |
| 块对角掩码 | "打包掩码" | 将每张图像的块限制为只在自身内部注意力计算，不聚合相邻图像的注意力掩码 |
| AnyRes | "LLaVA-NeXT 分块拼接" | 将高分辨率图像分割为固定大小格的网格加一个全局缩略图；用固定编码器对每个格编码 |
| NaFlex | "SigLIP 2 原生灵活" | 单个 SigLIP 2 检查点，推理时无需重新训练，支持 256/729/1024 词元预算 |
| M-RoPE | "多模态 RoPE" | 三维旋转位置编码（时间、行、列），无需位置表即可处理任意 H、W、T |
| cu_seqlens | "FlashAttention 打包" | FlashAttention 变长路径使用的累积长度张量，替代密集块对角掩码 |
| min_pixels / max_pixels | "分辨率边界" | Qwen2.5-VL 每请求的参数，限制极小或极大输入的词元数量 |
| 视觉词元预算 | "每张图像多少词元" | 每张图像产生的块词元大致数量；决定 LLM 的提示预算和注意力计算成本 |
