# Qwen-VL 系列与动态帧率视频

> Qwen-VL 系列——Qwen-VL（2023）、Qwen2-VL（2024）、Qwen2.5-VL（2025）、Qwen3-VL（2025）——是 2026 年最具影响力的开放视觉-语言模型谱系。每一代都做出了一个决定性的架构赌注，并在十二个月内被开放生态系统其他成员所效仿：通过 M-RoPE 实现原生动态分辨率、带绝对时间对齐的动态帧率采样、ViT 中的窗口注意力，以及结构化智能体输出格式。到 Qwen3-VL，配方已经稳定：一个带原生长宽比输入的 2D-RoPE-ViT 编码器，一个投射到大型 Qwen3 语言基础的 MLP 投影器，以及将 OCR、定位和智能体行为作为一等训练目标的训练阶段。本课按时间顺序解读这个系列，帮你理解每个参数为何在其所在位置。

**类型：** 学习
**编程语言：** Python（标准库，M-RoPE 编码器 + 动态帧率采样器）
**前置知识：** Phase 12 · 06（patch-n'-pack）
**预计时间：** 约 120 分钟

## 学习目标

- 计算 M-RoPE 的三轴旋转（时间、高度、宽度），并解释为什么三个轴都是必需的。
- 为一段视频选择动态帧率采样策略，并权衡每秒词元数与事件检测准确率。
- 按顺序说出 Qwen-VL 四代升级内容及各自实现了什么。
- 配置 Qwen2.5-VL 风格的 JSON 智能体输出格式，并从 VLM 回复中解析结构化工具调用。

## 问题背景

Qwen-VL 于 2023 年 8 月发布，是对 LLaVA-1.5 和 BLIP-2 的直接回应。Qwen 团队锁定的差距有三个：分辨率、视频和结构化输出。

**分辨率**：LLaVA-1.5 运行在 336×336。对照片够用，但对中文发票或密集的电子表格截图毫无用处。Qwen-VL 的第一个创新是 448×448 和定位边界框输出，让模型能够指向目标。

**视频**：Video-LLaMA 将逐帧编码器叠加后送入 LLM，对短片段有效，但对以时间轴为核心信号的多分钟视频无能为力。Qwen 团队想要一个能理解时间的单一编码器。

**结构化输出**：LLaVA 输出自由格式文本，而智能体需要 JSON。Qwen-VL 在显式 JSON 输出格式上进行了训练，包括边界框坐标的文本形式。

每一代 Qwen-VL 都在这三个轴中的一个上进行扩展。

## 核心概念

### Qwen-VL（2023 年 8 月）

第一代：OpenCLIP ViT-bigG/14 作为编码器（25 亿参数）、类 LLaMA 的 Q-Former（单步，256 个查询）、Qwen-7B 基础模型。贡献：

- 448×448 分辨率（当时开放 VLM 的 SOTA）。
- 定位：在带有显式坐标词元输出的图文配对上训练。"猫在 <box>(112, 204), (280, 344)</box>。"
- 从一开始就进行中英文双语训练。

当时的基准：英文上与 GPT-4V 相当，中文上占主导地位。定位监督是真正的亮点。

### Qwen2-VL（2024 年 9 月）——M-RoPE 与原生分辨率

Qwen2-VL 用原生动态分辨率 ViT 编码器替换了固定分辨率 + Q-Former 的组合。关键变化：

- **原生动态分辨率**：ViT 接受能被 28 整除的任意 H×W（Patch 14，2×空间合并）。一张 1120×672 的图像（40×24 合并块）产生 960 个视觉词元。无缩放，无分块拼接，无缩略图。
- **M-RoPE（多模态 RoPE）**：每个词元携带三维位置（t, h, w）而非一维。图像 t=0，视频 t=帧索引。RoPE 按每个轴的频率旋转查询/键向量，无需位置嵌入表。
- **MLP 投影器**：去掉 Q-Former，在合并后的块词元上使用 2 层 MLP。
- **动态帧率视频**：视频默认以 1–2 FPS 采样，但模型接受任意帧数。

结果：Qwen2-VL-7B 在多个多模态基准上与 GPT-4o 持平，并在 DocVQA 上超越（94.5 vs 88.4）。架构变化是决定性的举措。

### Qwen2.5-VL（2025 年 2 月）——动态帧率 + 绝对时间

Qwen2.5-VL 的重大变化在于视频。动态帧率不仅仅是"需要时采样更多帧"，该论文对此进行了正式定义：

- **绝对时间词元**：使用实际时间戳而非位置索引（帧 0、1、2...），时间戳词元 `<time>0.04</time>` 与帧词元交错出现。"在 0:04，猫跳了起来。"
- **动态帧率**：慢速片段用 1 FPS，动作场景用 4+ FPS，由用户或训练者选择，M-RoPE 自适应调整。
- **ViT 中的窗口注意力**：空间注意力在块内进行局部窗口化（提升吞吐量）；每隔几层有一次全局注意力。
- **显式 JSON 输出格式**：在工具调用数据上训练：`{"tool": "click", "coords": [380, 220]}`，开箱即用，智能体就绪。
- **MRoPE-v2 缩放**：位置随最大输入大小缩放，确保 10 分钟视频不会超出频率范围。

基准表现：Qwen2.5-VL-72B 在大多数视频基准上超越 GPT-4o，在文档理解上与 Gemini 2.0 相当，并创造了 GUI 定位的开放模型 SOTA（ScreenSpot：84% vs GPT-4o 的 38%）。

### Qwen3-VL（2025 年 11 月）

Qwen3-VL 是整合性的增量升级，而非重新发明：更大的 LLM 骨干（Qwen3-72B）、更多训练数据、改进的 OCR、通过 Qwen3 "思考模式"实现更强推理。ViT 和 M-RoPE 保持不变，论文重点是数据和训练改进而非架构。

系列总结：到 2025 年，Qwen-VL 架构已经稳定。后续版本扩展计算和数据，而非基本单元。

### M-RoPE 的数学原理

经典 RoPE 通过配对坐标将维度为 `d` 的查询向量 `q` 按位置 `m` 旋转：

```
q_rot[2i]   = q[2i]   * cos(m * theta_i) - q[2i+1] * sin(m * theta_i)
q_rot[2i+1] = q[2i]   * sin(m * theta_i) + q[2i+1] * cos(m * theta_i)
theta_i     = 10000^(-2i/d)
```

M-RoPE 将隐藏维度分为三段。设 `d = 96`，分配 32 维给时间，32 维给高度，32 维给宽度。每个维度段按各自的轴位置旋转。位置为 (t=5, h=10, w=20) 的块，其三个维度段分别应用旋转 `R_t(5)`、`R_h(10)`、`R_w(20)`。

文本词元使用 `t=文本索引, h=0, w=0`（或归一化选择），保持兼容性。视频帧使用 `t=帧时间, h=行, w=列`。单图像使用 `t=0`。

优势：一种位置编码无需分支代码或不同位置表，即可统一处理文本、图像和视频。

### 动态帧率采样逻辑

给定时长 `T` 秒的视频和目标词元预算 `B`：

1. 计算可负担的最大帧率：`fps_max = B / (T * tokens_per_frame)`。
2. 从 `{1, 2, 4, 8}` 中选择满足 `fps <= fps_max` 的目标帧率。
3. 若运动激烈（光流启发式或用户显式请求），选更高帧率；若运动缓慢，选更低帧率。
4. 按所选帧率均匀采样，帧之间插入 `<time>t</time>` 词元。

Qwen2.5-VL 隐式训练这一逻辑；推理时用户通过 `fps` 参数控制。一段 60 秒的动作场景在 4 FPS、每帧 81 词元下 = 19440 词元，在 32k 上下文中可管理。

### 结构化智能体输出

Qwen2.5-VL 的智能体训练明确针对结构化工具调用：

```
{
  "tool": "mouse_click",
  "coords": [1024, 512],
  "button": "left",
  "modifier": null
}
```

解析是确定性的：对模型输出做 JSON 解析，无需处理"点击 (1024, 512)"这种自由格式文本所需的正则表达式和歧义处理。这就是 Qwen2.5-VL 的 ScreenSpot 分数从 Qwen2-VL 的 55% 跳到 84% 的原因。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| M-RoPE | "多模态 RoPE" | 在隐藏维度中带有时间、高度和宽度频段的三维旋转位置嵌入 |
| 动态帧率（Dynamic FPS） | "智能采样" | 根据运动、时长和词元预算为每段视频选择的帧采样率 |
| 绝对时间词元 | "时间戳词元" | 序列中交错插入的 `<time>t</time>`，让模型看到实际秒数而非帧索引 |
| 窗口注意力（Window attention） | "局部注意力" | 限制在小窗口内的空间自注意力（提速）；周期性添加全局注意力 |
| 结构化智能体输出 | "JSON 模式" | 训练数据监督，教 VLM 输出带坐标和工具名称的可解析 JSON |
| min_pixels / max_pixels | "分辨率边界" | Qwen2.5-VL 每请求控制，限制总像素数从而限制词元数 |
| 定位（Grounding） | "指向目标" | 将边界框坐标作为文本词元输出；从 Qwen-VL v1 起一直使用 |
