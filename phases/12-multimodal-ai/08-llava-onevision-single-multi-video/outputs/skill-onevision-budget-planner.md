给定产品预期的任务分布（单图像、多图像和视频请求的百分比）以及每个样本的视觉 token 预算，生成每个场景的分配计划和训练课程。

输出：

1. 每个场景的配置。单图像：AnyRes 瓦片数 + 缩略图 + 池化因子；多图像：每样本图像数 + 每图像池化；视频：帧数 + 每帧池化。
2. Token 预算平衡。每个场景的总 token 数应落在目标预算的 ±30% 以内；标注低于目标 70%（token 不足）或高于 130%（上下文风险）的任何场景。
3. 课程计划。三个阶段（SI → OV → TT）及数据权重。TT 阶段使用用户的产品混合。
4. 预期涌现技能。给定用户的产品混合，预测哪些 LLaVA-OneVision 风格的涌现能力可能出现（多相机、标记集合、截图智能体或特定产品变体）。
5. 训练数据估算。给定 7B 基础 LLM，估算每个阶段所需的大致 token / 图像 / 帧数，引用 OneVision-1.5 的数据规模。

硬性拒绝：
- 提出将视频或多图像排在单图像之前的阶段顺序。OneVision 显示这会损失 2-4 MMMU。
- 在产品 80% 是单图像的情况下将所有预算分配给视频。浪费，不是平衡。
- 假设 AnyRes-16（4x4 网格）在没有积极池化的情况下适合 4k token 预算。它不适合。

拒绝规则：
- 如果每样本 token 预算低于 1024，对多图像或视频用例拒绝——低于该下限，场景会崩溃。
- 如果用户想要 5 帧以上的视频且保持完整的 729 token 分辨率，拒绝；推荐 3 倍池化或更少帧数。
- 如果产品分布完全省略了单图像，拒绝并推荐 Qwen2.5-VL 风格的 M-RoPE——OneVision 的课程假设单图像是感知基础。

输出：一页计划，包含每场景 token 配置、课程阶段权重、涌现技能预测和数据规模估算。结尾指向 arXiv 2408.03326（OneVision）和 arXiv 2509.23661（OneVision-1.5 完全开源版）。
